# 局域网分布式在线考试系统 - 产品需求文档 (PRD)

## 1. 产品概述和目标用户

### 1.1 产品概述

本项目是一款面向局域网环境的桌面端在线考试系统。系统旨在解决网络环境不稳定、甚至无公网连接情况下的数字考试需求。通过局域网直连、本地数据缓存、自动容灾及分布式部署等技术机制，确保考试过程的绝对稳定和数据的安全。

### 1.2 目标用户

* **教师/考务人员**：负责配置考试参数、导入题库、管理考试进度、监考、批阅试卷及导出成绩报告。
* **学生/考生**：在受控的桌面环境下安静答题，需要简单直观、防网络中断、防作弊的沉浸式考试体验。

## 2. 详细的功能列表

### 2.1 教师端功能

* **考试管理**
  * 新建/编辑考试：设置考试名称、考试起止时间、及格分数、及考试须知的富文本说明。
  * 状态控制：手动发卷、开始考试、暂停考试、结束考试，或设置定时自动执行。
* **题库与分值管理**
  * 模板导入：支持下载 Excel/CSV 模板，按规范一键导入试题。
  * 题型支持：单选题、多选题、判断题、填空题、简答题。
  * 分值与顺序：统一定义每题分值，支持题干及选项的随机打乱。
* **实时监考大屏**
  * 实时监控所有已连接考生的状态（在线、离线断网、答题进度、异常警告）。
  * 强制交卷：考务人员可针对某个异常学生执行强制收卷。
* **分布式与高可用容灾**
  * 组建集群：同一局域网内可打开多个教师端并形成集群（Master-Slave 模式）。
  * 自动心跳与切换：当主控教师端主机宕机或断网时，备用教师端自动接管考试网络，保障考试不中断，数据不丢失。
* **成绩与统计**
  * 自动评分：客观题（选择、判断、填空）在考后系统自动比对答案打分。
  * 手工阅卷：主观题（简答题）汇总至教师端，由教师手工批阅打分。
  * 成绩报告：生成本次考试的总成绩单及班级正确率统计，支持导出 Excel。

### 2.2 学生端功能

* **登录与准备**
  * 局域网自动发现：打开应用自动搜索局域网内的教师端发起的考试，无需手动输入 IP。
  * 身份验证：输入学号/姓名或特定考试码进入候考室。
* **考试作答**
  * 倒计时与进度墙：顶部醒目显示剩余时间，右侧或侧边栏显示题号矩阵（已答、未答、标记重看）。
  * 弱网无感重连：考试期间若局域网波动断开，本地继续缓存答题数据，网络恢复后后台静默重传与同步。
* **防作弊机制**
  * 全屏锁定：进入考试后自动全屏并限制系统快捷键（如 Alt+Tab，Win 键等）。
  * 自动录屏：考试期间后台自动录下屏幕画面，考试结束后或过程切片静默上传至教师端。

## 3. 功能优先级（MVP 和后续版本）

### 3.1 V1.0 - MVP 版本（最小可行产品）

**目标：跑通局域网考试的核心闭环。**

* 教师端创建考试、基础 Excel 题库导入。
* 单机版教师端服务（暂不带分布式容灾）。
* 局域网 UDP/TCP 组播发现机制。
* 学生端登录、客观题作答、断网本地缓存与重连同步。
* 客观题自动评分及成绩导出。

### 3.2 V1.5 - 进阶完善

**目标：强化安全性与考试严肃性。**

* 加入所有主观题（简答题）支持及教师手工阅卷工作流。
* 客户端防作弊限制（全屏锁定、快捷键屏蔽）。
* 学生端自动录屏及视频切片按需上传。

### 3.3 V2.0 - 高级稳定架构

**目标：解决高可用与大规模考场调度。**

* 教师端分布式部署上线（多节点状态同步与自动故障转移 failover）。
* 更详尽的考情分析看板和错题统计。

## 4. 界面设计要求

* **整体风格**：清爽、专业，以减少视觉疲劳的色彩为主（如主色调使用蓝色/蓝灰，辅助状态体现使用红绿金等）。
* **教师端设计**：
  * 面板式（Dashboard）布局，左侧导航。
  * 监考页面需大屏数据可视化设计，卡片形式展示每个考生的连接状态与进度。
  * 题目导入阶段需要有清晰的验证错误提示（如第X行格式错误）。
* **学生端设计**：
  * 极简沉浸式布局（沉浸模式）。
  * 字体清晰、行距适中，确保长时间读题不疲劳。
  * 右下角常驻“交卷”按钮，要有防误触确认交互（如滑动解锁交卷或确认弹窗）。

## 5. 技术栈建议（推荐 Tauri）

基于目前的需求描述，**Tauri** 是构建此桌面应用的最优选择（体积小、性能优、原生系统级API调用方便）：

* **底层及后端 (Tauri Core - Rust语言)**：
  * **网络通信**：使用 Rust `tokio` 进行高性能异步并发处理；使用 mDNS/Bonjour (`mdns-sd`) 实现局域网节点自动发现；WebSocket (`tungstenite`) 或 gRPC 作长链接状态同步。
  * **分布式状态**：Teacher 端可引入轻量级基于 Raft 原理的状态机或简单的 `sqlite` 实时文件同步确保主备切换。
  * **本地存储**：学生端离线存储可使用 `sled` 或 `sqlite`，避免数据丢包。
  * **录屏与系统控制**：利用 Rust 调用操作系统底层 API （如 `scrap` 或 `ffmpeg-sys` 库）实现静默录屏和系统级热键拦截。
* **前端渲染 (Webview)**：
  * **框架**：Vue 3 或 React 18。
  * **路由及状态管理**：Vue Router + Pinia 或 React Router + Zustand。
  * **UI 组件库**：Element Plus (Vue) 或 Ant Design (React)，满足后台表格和表单的重度需求。
  * **构建工具**：Vite（极致的冷启动与热更新体验）。

## 6. 非功能性需求（性能、安全等）

### 6.1 性能需求 (Performance)

* **并发处理**：单台常规性能的教师机（如 i5 处理器，8G内存）应当能同时支撑至少 **100-200 台** 局域网学生端的心跳维系和并发交卷。
* **数据承载**：考试题库应能顺畅加载至少千题级别，页面渲染延迟不高于 100ms。
* **弱网容忍度**：在网络丢包率高达 30% 或时不时断开长达数十分钟的情况下，学生端应用不可卡死，必须保障流畅作答。

### 6.2 安全需求 (Security)

* **防破解篡改**：本地缓存的试题以及答题记录需使用 AES 加密存储，防止学生通过修改本地文件作弊。
* **请求签名**：局域网通信时，指令带有时间戳与请求哈希（JWT 方案等），防止恶意抓包及重放攻击（如提前交卷/截获试题）。
* **进程保护**：学生端在考试过程中检测到类似虚拟机操作、特定抓包软件运行时应向教师端告警。

### 6.3 兼容性需求 (Compatibility)

* 主要支持 Windows 10 及 Windows 11 操作系统（大多数学校机房环境）。
* 可不修改核心业务逻辑直接编译为 macOS 和 Linux 桌面版（Tauri 自带跨平台特性）。
