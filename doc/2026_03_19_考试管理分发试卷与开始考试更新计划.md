# 考试管理分发试卷与开始考试更新计划（2026-03-19）

## 目标

在教师端“考试管理”页面打通“分发试卷”和“开始考试”两条链路，形成以下闭环：

1. 教师端在考试管理页选择考试后，可对当前考试下已分配学生分发试卷。
2. 教师端通过现有 TCP 控制链路逐台向学生端下发试卷快照。
3. 学生端收到试卷后，将考试会话与试卷快照保存到本地数据库。
4. 学生端在收到试卷后进入候考态，但在收到“开始考试”指令前不能显示试卷开始作答。
5. 教师端点击“开始考试”后，通过现有 WebSocket 链路向学生端广播开始考试指令。
6. 学生端收到开始考试指令后，切换到正式考试态，并从本地快照加载试卷开始答题。

本次先完成发卷与开考闭环，不实现学生答题过程中每次选项变化同步到教师端。

---

## 范围

### 本次纳入范围

1. 教师端考试管理页的“分发试卷”业务动作。
2. 教师端考试管理页的“开始考试”业务动作。
3. 教师端 Rust 后端按考试批量组装试卷快照并逐台下发。
4. 学生端 TCP 控制服务接收试卷并落库到本地考试相关表。
5. 学生端 WebSocket 客户端接收开始考试消息并切换考试状态。
6. 学生端主页与考试页根据本地考试状态切换展示候考态与开考态。
7. 必要的前后端状态校验、按钮禁用与错误反馈。

### 本次不纳入范围

1. 不实现学生答题过程中的实时答案同步。
2. 不实现自动交卷、强制交卷、暂停考试等扩展控制动作。
3. 不实现复杂的断线重连、失败重试队列或补偿机制。
4. 不重构现有加密方案，只在现有快照结构内完成可用链路。
5. 不新增教师端成绩回收、阅卷、报告相关能力。

## 现状结论

### 教师端现有能力

1. 教师端考试管理页已存在“分发试卷”和“开始考试”按钮，但当前仅更新考试状态，没有真正发卷或发开始指令。
2. 教师端前端入口位于 apps/teacher/src/pages/ExamManage/index.tsx 与 apps/teacher/src/hooks/useExamManage.ts。
3. 教师端 Rust 已具备按考试批量连接学生设备的模板，可复用为按考试批量发卷的入口，现有模板位于 apps/teacher/src-tauri/src/controllers/student_exam_controller.rs。
4. 教师端已具备 TCP 控制客户端，可向学生端发送 APPLY_TEACHER_ENDPOINTS 类型消息，链路位于 apps/teacher/src-tauri/src/network/student_control_client.rs。
5. 教师端已具备 WebSocket 服务与学生连接状态管理骨架，适合作为开始考试广播通道。
6. 教师端题库与试卷数据来源已经存在，考试信息可从 exam_service 获取，题目列表可从 question_service 获取。

### 学生端现有能力

1. 学生端已具备 TCP 控制服务 control_server，可接收教师端单请求单回执消息。
2. 学生端已具备教师地址落库与收到地址后主动连接教师端 WebSocket 的逻辑。
3. 学生端数据库已经定义 exam_sessions 与 exam_snapshots 两张表，但运行时还没有真正用于收卷与开考闭环。
4. 学生端 WebSocket 客户端当前只负责发送心跳，收到教师端消息后仅打印日志，尚未处理开始考试消息。
5. 学生端前端当前直接进入模拟考试页，考试内容仍来自本地临时数据，不依赖本地数据库中的试卷快照。

### 关键问题

1. 当前“分发试卷”只是把考试状态改成 published，未向学生端发送任何试卷数据。
2. 当前“开始考试”只是把考试状态改成 active，未通知学生端切换到开考态。
3. 学生端虽然已有考试相关表结构，但缺少服务层与命令层来承接落库、查询与状态切换。
4. 学生端前端还没有“候考态”与“开考态”的页面切换逻辑。

---

## 关键设计决策

1. 分发试卷走现有 TCP 控制链路，复用 control_server 的单请求单回执模式。
2. 开始考试走现有 WebSocket 链路，使用教师端统一广播的方式触发学生端开考。
3. 考试状态变更必须在业务动作成功后执行，避免教师端先改 published 或 active 但学生端实际未收卷或未开考。
4. 学生端在收到试卷后只进入候考态，不允许提前展示试卷内容开始作答。
5. 学生端题目展示数据来源切换为本地 exam_snapshots，不再使用前端临时模拟题目。
6. 本次以闭环可用为优先目标，不把答案实时同步耦合进来。

## 目标交互流程

### 一、分发试卷流程

1. 教师进入“考试管理”页面。
2. 教师选择一场考试。
3. 页面查询当前考试详情、题目列表与已分配学生设备列表。
4. 教师点击“分发试卷”。
5. 教师端后端按考试维度逐台组装试卷快照 payload。
6. 教师端通过 TCP 控制客户端连接学生端 control_port，发送分发试卷请求。
7. 学生端 control_server 接收请求后，进入考试会话与试卷快照落库事务。
8. 学生端落库成功后返回 ACK。
9. 教师端汇总设备级成功与失败结果。
10. 所有成功条件满足后，教师端将考试状态切换为 published。

### 二、开始考试流程

1. 教师确认当前考试已完成发卷。
2. 教师点击“开始考试”。
3. 教师端通过 WebSocket 向当前考试范围内学生广播开始考试消息。
4. 学生端收到开始考试消息后，更新本地 exam_sessions 状态为 active，并写入 started_at、ends_at 等字段。
5. 学生端前端检测到本地考试状态变为 active 后，读取 exam_snapshots 中的试卷快照。
6. 学生端切换到正式考试页面并允许答题。
7. 教师端广播成功后，将考试状态切换为 active。

## 推荐实施顺序

### 阶段一：协议与后端入口先行

1. 定义发卷 TCP 协议与开考 WebSocket 协议。
2. 教师端补齐批量发卷命令与开始考试命令。
3. 学生端补齐 control_server 收卷分支与 ws_client 开考分支。

### 阶段二：学生端本地持久化闭环

1. 实现 exam_sessions 与 exam_snapshots 的写入与查询服务。
2. 验证收卷后本地库中可查到会话和快照。
3. 验证收卷后仍处于候考态。

### 阶段三：教师端页面与状态闭环

1. 考试管理页接入真实发卷与开考命令。
2. 补充按钮禁用、加载态、错误提示与设备级结果反馈。
3. 保证动作成功后再切考试状态。

### 阶段四：学生端页面接入本地快照

1. 以本地 exam_sessions 状态驱动页面。
2. 以本地 exam_snapshots 数据驱动题目展示。
3. 候考态与正式考试态分开展示。

### 阶段五：联调与回归

1. 先复用现有连接考生设备流程建立教师端与学生端连接。
2. 再执行分发试卷，确认落库成功。
3. 最后执行开始考试，确认页面切换成功。

## 验收标准

1. 教师端点击“分发试卷”后，不再只是更新考试状态，而是能向学生端实际发送试卷数据。
2. 学生端收到试卷后，本地 exam_sessions 与 exam_snapshots 有对应数据记录。
3. 学生端收到试卷后仍处于候考态，不能直接开始答题。
4. 教师端点击“开始考试”后，学生端能收到开始指令并切换到正式考试态。
5. 学生端正式考试页展示的数据来源于本地快照，而非前端临时模拟数据。
6. 教师端只有在业务动作成功后才把考试状态切换到 published 或 active。
7. 本次不要求学生答题时实时把每次选项变化同步到教师端。
