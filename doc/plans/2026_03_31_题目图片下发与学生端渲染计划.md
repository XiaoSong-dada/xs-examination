# 教师端与学生端题目图片下发与渲染计划（2026-03-31）

## 1. 目标

在当前“连接考生设备 -> 分发试卷 -> 学生端本地落库 -> 开始考试”的既有闭环上，补齐“题干图片 + 选项图片”的完整能力，满足以下目标：

1. 教师端题库能够描述题干图片与选项图片，而不是只支持纯文本题目。
2. 图片附件在数据库中以本地地址形式保存，而不是以内联二进制或 data URL 长字符串保存。
3. 试卷元数据仍可沿用现有发卷闭环落到学生端本地。
4. 图片二进制不继续塞进现有 `questions_payload` 大字符串，而是通过现有学生 WebSocket 链路直传到学生端。
5. 学生端题目页不沿用当前预留做小修补，而是按“题干块 + 选项组”重做图片型组件。
6. 发卷成功的判定不再只看“试卷元数据落库”，还要覆盖图片资源是否同步完成。

本计划优先实现“图片可导入、可下发、可缓存、可渲染”的闭环，不在本次扩展音频、视频、富文本编辑器和复杂混排排版能力。

---

## 2. 范围

### 2.1 本次纳入范围

1. 教师端题目结构扩展：支持题干图片、选项图片。
2. 教师端题库导入格式升级：支持题目数据与图片资源一起导入。
3. 教师端发卷链路扩展：题目元数据与图片资源分离下发。
4. 教师端 WebSocket 协议扩展：新增图片资源同步消息与 ACK。
5. 学生端本地图片资源落库或落缓存。
6. 学生端题目组件重制，支持文字选项与图片选项混排。
7. 学生端试卷解析与题目页渲染升级。
8. 分发结果与最小验收口径调整。

### 2.2 本次不纳入范围

1. 不在本次把整条发卷链从 TCP 控制请求整体改成纯 WebSocket 发卷。
2. 不在本次实现题目图片在线编辑器。
3. 不在本次支持图片标注、裁剪、缩放编辑、拖拽排序等增强能力。
4. 不在本次支持音频、视频、PDF 等多媒体题干。
5. 不在本次重构 `packages/shared-types` 为现行协议单一事实来源。

## 3. 当前现状

### 3.1 已有发卷链路现状

基于现有代码，当前教师端发卷链路是：

1. 教师端前端 `pages/ExamManage -> hooks/useExamManage -> services/studentService.ts` 调用 `distribute_exam_papers_by_exam_id`。
2. 教师端 Rust `student_exam_service::distribute_exam_papers_by_exam_id` 读取考试与题目后，把整份试卷序列化成 `questions_payload`。
3. 教师端通过 `student_control_client::distribute_exam_paper` 按设备 IP 直连学生端控制端口。
4. 学生端 `control_server.rs` 在 `DISTRIBUTE_EXAM_PAPER` 分支中调用 `ExamRuntimeService::upsert_distribution`，把 `exam_sessions` 与 `exam_snapshots` 落到本地数据库。

这条链路当前是“试卷元数据落库链”，不是 WebSocket 直推链。

### 3.2 题目结构现状

当前题目结构只覆盖纯文本题目：

1. 教师端 `questions` 表只有 `content`、`options`、`answer`、`score`、`explanation`，没有图片专用字段。
2. 教师端 `question_schema.rs`、`question_service.rs`、`question_repo.rs` 也只透传上述文本字段。
3. 教师端发卷时构造的 `questions_payload` 只包含 `id/seq/type/content/options/score/explanation`。

### 3.3 学生端渲染现状

学生端当前虽然已经有零散图片展示预留，但这次方案不以这些预留为基础继续追加，而是按新的题目结构重新设计组件：

1. `pages/Exam/index.tsx` 会调用 `ImageList` 渲染 `currentQuestion.images`。
2. `store/examStore.ts` 解析 `questions_payload` 时，当前把每道题的 `images` 强制写成空数组。
3. `AnswerList` 目前只能渲染选项文本，不能渲染选项图片。

这意味着当前学生端只有展示占位，没有真正的数据来源，也不具备“选项类型切换 + 选项文字下挂图片”的目标组件结构。

### 3.4 WebSocket 现状

当前学生端与教师端之间已经有长期 WebSocket 连接，但职责集中在运行态消息：

1. 已有消息类型主要是 `EXAM_START`、`FINAL_SYNC_REQUEST`、`EXAM_END`、`ANSWER_SYNC`、`ANSWER_SYNC_ACK`、`HEARTBEAT`。
2. 现有 WebSocket 信封是 JSON 文本消息，不是二进制帧协议。
3. 教师端已经按 `student_id` 维护在线学生到 WS peer 的映射，因此具备“按学生定向推送资源”的基础。

## 4. 核心决策

### 4.1 图片采用“元数据继续发卷，二进制改走 WebSocket”方案

本次不建议把图片直接塞进现有 `questions_payload`，而采用两段式方案：

1. 题目元数据继续沿用当前 `DISTRIBUTE_EXAM_PAPER` 控制链路落库。
2. 图片二进制通过现有学生 WebSocket 连接分块传入学生端。

原因：

1. 当前学生端控制服务对 `DISTRIBUTE_EXAM_PAPER` 采用单次大报文读取，继续把图片直接并入会快速放大 payload 体积。
2. `questions_payload` 与 `exam_snapshots` 当前定位是试卷快照，不适合作为大体积图片二进制容器。
3. 用户明确希望图片通过 WebSocket 传入学生端。
4. 现有教师端已经具备按 `student_id` 精准推送到在线学生的 WS 基础能力。

### 4.2 题目结构采用“文本字段 + 本地地址字段”方案

本次不建议在题干或选项里直接塞 HTML，也不建议直接把整张图作为 data URL 固化进题目主字段。推荐采用“文本字段 + 图片本地地址字段”的结构：

1. 题干文本继续保留在 `content`。
2. 题干图片新增 `content_image_paths` 字段，保存本地地址数组。
3. `options` 从当前的简单 JSON 扩展为对象数组；每个选项必须带 `optionType`，图片型选项同时保存文字与图片本地地址。

推荐的运行态结构：

```json
{
  "id": "question-id",
  "seq": 1,
  "type": "single",
  "content": "下面图片对应哪种设备？",
  "contentImagePaths": [
    "C:/Users/.../teacher-assets/question-1/stem-1.png"
  ],
  "options": [
    {
      "key": "A",
      "optionType": "text",
      "text": "路由器",
      "imagePaths": []
    },
    {
      "key": "B",
      "text": "交换机",
      "optionType": "text_with_image",
      "imagePaths": [
        "C:/Users/.../teacher-assets/question-1/option-b-1.png"
      ]
    }
  ]
}
```

这个结构的好处是：

1. 对纯文本题目兼容。
2. 选项可以明确声明自己的展示类型，组件设计边界更清晰。
3. 图片附件可以直接以本地地址形式持久化。

### 4.3 本地地址按端内生效，不直接跨端复用

这里需要明确一个边界：

1. 教师端数据库里保存的是教师端本地地址。
2. 学生端收到图片并落盘后，学生端数据库里保存的是学生端本地地址。
3. 教师端本地绝对路径不能直接下发给学生端当作最终可渲染地址使用。

因此，本次方案不应理解为“教师端把本机路径原样发给学生端就结束”，而应理解为：

1. 教师端出题或导入时，先把图片复制到教师端受控资源目录，再把教师端本地地址写入数据库。
2. 发卷时，教师端依据这个本地地址读取文件内容，通过 WebSocket 传给学生端。
3. 学生端收到后保存到自己的受控资源目录，再把学生端本地地址写入学生端本地数据库或试卷快照。

这样才同时满足“数据库保存本地地址”与“学生端可实际渲染”两个条件。

### 4.4 WebSocket 图片同步先沿用 JSON 文本信封 + 分块 base64

当前 WebSocket 传输层已经稳定使用 JSON 文本消息。本次优先不重写底层 transport 为二进制帧协议，而采用：

1. 新增图片同步消息类型。
2. 每个 chunk 通过 JSON 文本消息发送。
3. chunk 内携带 base64 片段、序号、总片数与资源标识。

原因：

1. 可以复用现有 `build_message / encode_message / decode_value_message`。
2. 改动集中在协议与业务处理层，不需要先大改 transport。
3. 对当前仓库更符合最小改动原则。

后续若图片规模显著增加，再单独计划把图片同步切到 WS 二进制帧。

### 4.5 发卷成功口径改为“元数据 ACK + 图片 ACK”

当前发卷成功只以 `DISTRIBUTE_EXAM_PAPER_ACK` 为准。本次建议调整为：

1. 先收到学生端 `DISTRIBUTE_EXAM_PAPER_ACK`，说明题目元数据已落库。
2. 再收到该学生全部必需图片资源的同步完成 ACK，说明题目图片已可用。
3. 两者都成功，才把该学生目标计为完整成功。

否则应返回“试卷元数据已落库，但图片同步失败/未完成”的部分成功结果。

## 6. 实施方案

### 6.1 第一阶段：定义题目图片数据模型

目标：先把“图片是什么”定义清楚，再做传输。

计划：

1. 教师端 `Question` DTO 增加 `content_image_paths`。
2. 教师端 `options` 标准化为 JSON 数组对象，每个选项至少包含：
  - `key`
  - `text`
  - `optionType`
  - `imagePaths`
3. 学生端运行态 `RuntimeQuestion` 同步增加：
  - `contentImagePaths`
  - `options[].optionType`
  - `options[].imagePaths`
4. 选项类型先收敛为两种：
  - `text`
  - `text_with_image`
5. 图片附件记录除本地地址外，仍建议附带元信息用于校验与传输：
  - `fileName`
  - `mimeType`
  - `sha256`
  - `size`

说明：

1. `content_image_paths` 建议通过 migration 新增为 `TEXT`，内容为 JSON 字符串。
2. `options` 仍沿用 JSON 字符串列，但对象结构升级。
3. 数据库存的是本地地址，但为了 WebSocket 传输与幂等，仍应保留摘要字段，不建议只存裸路径。

### 6.2 第二阶段：升级教师端导入格式

目标：让教师端有稳定的图片输入来源。

建议方案：

1. 教师端新增或改造出题入口，使题干图片和选项图片都可以通过本地文件选择器挂载。
2. 题库批量导入若继续保留 Excel 路径，可新增列：
  - `content_image_paths`
  - `option_a_type`
  - `option_a_image_paths`
  - `option_b_type`
  - `option_b_image_paths`
  - `option_c_type`
  - `option_c_image_paths`
  - `option_d_type`
  - `option_d_image_paths`
3. 无论是手工出题还是批量导入，最终都统一执行“复制到教师端受控资源目录，再把复制后的本地地址入库”。

原因：

1. 用户已明确希望数据库保存本地地址。
2. 直接保存用户原始选择的桌面路径不稳定，文件被移动后会失效。
3. 先复制到应用受控目录，再入库本地地址，才是可持续的本地地址方案。

教师端导入后处理：

1. 校验所选图片文件是否存在且格式可接受。
2. 将图片复制到教师端本地资源目录，例如 `app_data_dir/question-assets/`。
3. 把复制后的教师端本地地址写入题目结构。
4. 对图片型选项，除文字外同步保存该选项的 `optionType` 与 `imagePaths`。

### 6.3 第三阶段：发卷时把元数据与图片资源拆开

目标：保持现有发卷主链稳定，同时满足“图片通过 WebSocket 传给学生端”。

发卷拆分为两步：

1. 第一步：沿用现有 `DISTRIBUTE_EXAM_PAPER`，只发送题目元数据、题干图片地址字段和选项图片地址字段，不发送图片二进制本体。
2. 第二步：教师端针对发卷成功且已在线的学生，通过 WebSocket 发送该场考试需要的图片资源。

建议新增消息类型：

1. `PAPER_ASSET_MANIFEST`
2. `PAPER_ASSET_CHUNK`
3. `PAPER_ASSET_ACK`
4. `PAPER_ASSET_SYNC_DONE`

建议流程：

1. 教师端先下发题目元数据。
2. 学生端根据题干图片地址和选项图片地址字段，生成当前 session 的待同步图片清单。
3. 教师端通过 `PAPER_ASSET_MANIFEST` 告知总资源数与资源摘要。
4. 教师端逐个资源、逐片发送 `PAPER_ASSET_CHUNK`。
5. 学生端每个资源完成组包后校验 `sha256`，保存到学生端本地资源目录，并把学生端本地地址写回当前题目快照。
6. 学生端单资源完成后回 `PAPER_ASSET_ACK`。
7. 全部资源完成后，学生端回 `PAPER_ASSET_SYNC_DONE`。

### 6.4 第四阶段：学生端组件重做与可见渲染

目标：学生端不沿用当前预留做小补丁，而是直接按新的题目结构重做组件，并在图片落地后稳定显示。

组件方案：

1. 题目区拆成“题干块 + 选项组”两层结构。
2. 题干块负责展示题干文字和题干图片。
3. 选项组不再只是平铺文本按钮，而是新增选项内容组件。
4. 单个选项根据 `optionType` 选择渲染：
  - `text`：只展示选项文字。
  - `text_with_image`：先展示选项文字，再在该选项下方展示对应图片。
5. 也就是说，图片型选项的 UI 不是“图片替代文字”，而是“文字在上，图片在下”。

建议本地落点：

1. 图片二进制保存到学生端本地缓存目录，例如 `app_data_dir/paper-assets/{session_id}/`。
2. 学生端为每个资源记录本地状态：`pending / ready / failed`。
3. 学生端在资源落盘后，把学生端本地地址回写到题目快照或配套图片表中。
4. `get_current_exam_bundle` 或对应读取链路返回题目时，直接返回学生端本地地址，不再依赖教师端路径。

前端渲染改动：

1. `examStore.parseQuestions` 不再把 `images` 固定写空，而是解析题干图片地址与选项图片地址。
2. 当前 `ImageList` 与 `AnswerList` 不作为最终组件边界，允许重构或替换。
3. 新组件需要原生支持选项类型分流，不再把图片当作列表尾部附加信息。
4. 当图片未就绪时，页面明确显示“图片同步中”或“图片加载失败”，不要静默空白。

### 6.5 第五阶段：补齐重试与幂等

目标：避免大图同步因为短暂断线导致整场考试纸面不可用。

建议机制：

1. 图片资源以 `assetId + sha256` 作为幂等键。
2. 学生端若本地已存在同 `assetId` 且 `sha256` 一致，可直接跳过重复写入。
3. 教师端若收到学生端资源已存在 ACK，可跳过重复发送。
4. 若 WS 中断，允许在重新连接后继续对当前 session 补发未完成资源。
5. 若数据库中的本地地址所指文件缺失，应把状态标记为失效并允许重新同步，不把旧路径当作成功。

## 7. 分阶段计划

### 7.1 P1：题目结构与导入格式

1. 设计基于本地地址的题干图片字段与选项图片字段。
2. 设计升级后的 `QuestionOption` 结构，补齐 `optionType + text + imagePaths`。
3. 增加教师端 migration 与题目 DTO。
4. 改造出题/导入入口，统一把图片复制到教师端受控目录后再入库。

### 7.2 P2：发卷链路拆分

1. 修改教师端发卷 payload，发送题目文本与图片地址字段，但不发送图片二进制。
2. 修改学生端试卷落库逻辑，接受题干图片地址字段与选项图片地址字段，并记录待同步状态。
3. 教师端发卷结果中新增“图片同步状态”。

### 7.3 P3：WebSocket 图片同步

1. 扩展教师端/学生端 WS 协议。
2. 实现 manifest、chunk、ACK、完成态。
3. 实现失败重试与超时处理。

### 7.4 P4：学生端页面渲染

1. 重做题干块与选项组组件。
2. 支持文字选项与图片型选项两类展示。
3. 图片型选项采用“文字在上、图片在下”的布局。

## 8. 风险与边界

### 8.1 主要风险

1. 当前 WebSocket 是文本信封，图片使用 base64 分块会带来额外体积膨胀。
2. 若一场考试图片过多，发布耗时会明显高于纯文本试卷。
3. 当前教师端发卷成功口径需要改造，否则容易出现“试卷已发，但图片未齐”的伪成功。
4. 若批量导入继续保留 Excel 路径，教师端题库导入交互仍需同步扩展图片列与文件校验流程。
5. 如果只保存原始选择路径而不复制文件，后续文件被移动或删除会导致数据库中的地址失效。

### 8.2 边界约束

1. 不把图片标准答案信息透传给学生端。
2. 不把绝对文件路径暴露给学生端前端。
3. 不在本次把所有题目内容重构成通用富文本协议。
4. 不在本次用 `packages/shared-types` 统一两端协议；仍以两端实际 `schemas/types` 为准。
5. 教师端本地地址只作为教师端资源定位依据；学生端最终渲染必须使用学生端本地地址。

### 8.3 需要尽早确认的点

1. 批量导入是继续沿用 Excel 并填写图片路径，还是另行补一套更稳定的资源包导入方式。
2. 发布成功是否必须等待全部图片资源 ACK 后再标记考试 `published`。
3. 教师端是否还需要新增“手工出题并选择本地图片”的入口，而不是只覆盖批量导入。

## 9. 最小验收步骤

### 9.1 题库导入验收

1. 准备一条带题干图片与选项图片的题目数据，可来自手工出题或批量导入。
2. 教师端保存后，数据库中保存的是复制后的教师端本地地址，而不是原始外部路径。
3. 教师端本地资源目录能看到对应缓存文件。

### 9.2 发卷验收

1. 教师端连接目标学生设备。
2. 点击“分发试卷”。
3. 教师端返回结果中，既显示试卷元数据是否落库，也显示图片资源是否完成同步。

### 9.3 学生端渲染验收

1. 学生端收到试卷后，等待页或考试页不再只有纯文本。
2. 题干图片可见。
3. 带图片的选项可见，且布局为“选项文字在上、选项图片在下”。
4. 关闭学生端再打开后，图片仍可从本地恢复显示。

### 9.4 异常验收

1. 在图片同步过程中断开教师端 WS。
2. 学生端应进入“图片未完成”状态，而不是显示为空白成功。
3. 恢复连接后，可以继续补发未完成图片。
