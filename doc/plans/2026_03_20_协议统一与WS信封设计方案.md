# 协议统一与 WS 信封设计方案（2026-03-20）

## 目标

针对当前教师端与学生端同时使用 TCP 与 WebSocket 的现状，明确以下问题：

1. 当前是否必须统一协议。
2. TCP 与 WebSocket 在本项目中的优劣与适用边界。
3. 后续主业务协议应统一到哪一种传输方式。
4. 若统一到 WebSocket，需要怎样定义统一消息信封。
5. Rust 侧应优先采用哪些框架承接后续演进。

---

## 现状结论

结合现有文档与代码，当前并非无序混用，而是已经形成了两类通道：

1. TCP 控制链路。
2. WebSocket 实时链路。

当前职责分工如下：

1. 教师端通过 `student_control_client.rs` 向学生端发送 `APPLY_TEACHER_ENDPOINTS` 与 `DISTRIBUTE_EXAM_PAPER`。
2. 学生端通过 `control_server.rs` 接收上述 TCP 请求，并返回单次 ACK。
3. 学生端收到教师地址后，主动连接教师端 WebSocket。
4. 教师端通过 `ws_server.rs` 维护学生连接、心跳状态与定向消息发送能力。
5. 学生端通过 `ws_client.rs` 维持长连接、发送心跳，并接收 `EXAM_START` 等实时消息。

因此，当前结构本质上是：

1. TCP 负责单次控制请求。
2. WS 负责持续双向实时消息。

这套分工在短期内可运行，但随着控制动作、实时状态与同步事件变多，会带来明显维护成本。

---

## 当前双协议方案的主要问题

### 1. 业务语义分散

同一业务域的消息分散在两条协议中，例如：

1. 发卷走 TCP。
2. 开考走 WebSocket。
3. 心跳走 WebSocket。
4. 后续答案同步、状态上报、作弊告警也规划走 WebSocket。

结果是业务方需要同时理解两套入口、两套异常、两套路由与两套回执模型。

### 2. 基础能力重复建设

若继续扩展 TCP 控制链路，就需要不断补齐：

1. 请求关联。
2. ACK 规范。
3. 超时处理。
4. 错误码。
5. 幂等控制。
6. 重试策略。
7. 会话路由。

这些能力在长连接实时通道中本就更自然。

### 3. TCP 当前实现只适合短请求

现有 TCP 实现属于“短连接 + 完整 JSON + EOF/半关闭作为边界识别”的模式，适合：

1. 发一条请求。
2. 收一个回执。
3. 立即结束连接。

它不适合继续演进成统一业务总线，因为：

1. 不适合持续双向收发。
2. 不适合广播。
3. 不适合复用同一连接承载多种动作。
4. 后续要么继续堆特殊处理，要么需要重写成长度分帧协议。

### 4. 后续演进方向已经明显偏向 WS

从现有技术设计可见，以下消息本身都更适合长连接：

1. `HEARTBEAT`
2. `ANSWER_SYNC`
3. `STATUS_UPDATE`
4. `CHEAT_ALERT`
5. `FORCE_SUBMIT`
6. `EXAM_START`
7. `EXAM_PAUSE`
8. `EXAM_END`

这意味着系统未来增长的大多数业务消息，本质上都偏向 WebSocket，而不是偏向短连接 TCP。

## TCP 与 WebSocket 对比

### 一、TCP 的优点

1. 协议层级更低，开销更小。
2. 行为完全可控，便于实现自定义二进制协议。
3. 做单次请求/回执足够直接。
4. 对超大文件流或定制化帧协议更友好。

### 二、TCP 的缺点

1. 需要自行处理消息边界。
2. 需要自行处理心跳与保活。
3. 需要自行处理连接复用。
4. 需要自行处理广播与单播路由。
5. 需要自行处理 ACK、重试、幂等与请求关联。
6. 调试成本高于标准化的文本 WebSocket 消息。

### 三、WebSocket 的优点

1. 天然支持双向通信。
2. 天然支持长连接复用。
3. 非常适合教师端广播与学生端持续上报。
4. 更适合承载实时控制消息、状态消息与同步消息。
5. 可以统一消息信封，减少协议分裂。
6. 对当前项目的 Rust 与前端现有实现更友好。

### 四、WebSocket 的缺点

1. 比裸 TCP 多一层握手与封装。
2. 若所有业务都直接混发在一条连接中，后续需要消息分类和发送优先级管理。
3. 若学生端尚未建立连接，则教师端无法直接主动推送业务消息，仍需保留首次发现或引导机制。

---

## 是否必须统一协议

结论如下：

1. 不必为了形式上“只剩一种底层传输”而强行统一所有链路。
2. 但必须逐步统一业务协议模型与主业务通道。

更准确地说，本项目需要统一的是：

1. 业务消息模型。
2. 请求/响应/ACK 语义。
3. 路由字段。
4. 错误码与幂等规则。
5. 主业务消息的承载通道。

而不是立即要求所有发现、引导、配置、文件传输都彻底压成同一种底层协议。

---

## 建议的统一方向

## 结论

建议将主业务协议统一到 WebSocket。

不建议统一到裸 TCP。

### 原因

1. 本项目后续增长的消息类型天然更适合长连接实时通信。
2. 当前教师端已经具备 WebSocket 服务与学生连接映射能力。
3. 当前学生端已经具备 WebSocket 客户端、心跳与消息接收能力。
4. 若统一到 TCP，最终仍需自行重建连接管理、路由、ACK、重试与广播能力。
5. 统一到 WS 的迁移成本明显低于统一到裸 TCP。

### 对“首次连接”的处理建议

统一业务协议到 WebSocket，不代表所有事情都必须只靠 WS 完成。

首次发现或引导仍可保留以下轻量机制之一：

1. mDNS 自动发现教师端地址。
2. 手工配置教师端地址。
3. 轻量 HTTP 或轻量 TCP 的 bootstrap 通道。

即：

1. 首次发现可以不是 WS。
2. 一旦连接建立，主业务控制与实时消息统一走 WS。

---

## 为什么不建议统一到裸 TCP

如果统一到裸 TCP，后续很可能需要在应用层重新发明一套类似 WebSocket 的能力：

1. 消息分帧。
2. 消息类型路由。
3. 请求 ID。
4. ACK 与响应关联。
5. 心跳。
6. 重连。
7. 广播与单播。
8. 去重与幂等。

从工程收益看，这相当于为了解决“协议看起来统一”而主动承担更高的实现成本，收益不成比例。

---

## 推荐的目标架构

### 一、控制面与实时面统一为 WS

以下消息统一走 WebSocket：

1. `EXAM_PAPER_DISTRIBUTE`
2. `EXAM_START`
3. `EXAM_PAUSE`
4. `EXAM_END`
5. `FORCE_SUBMIT`
6. `HEARTBEAT`
7. `ANSWER_SYNC`
8. `STATUS_UPDATE`
9. `CHEAT_ALERT`

### 二、超大资源下载不强行走 WS

若后续试卷中包含大体积资源，例如：

1. 图片包。
2. 音频包。
3. 视频包。
4. 附件包。

则更合理的方式是：

1. WS 负责控制消息与下载指令。
2. HTTP 负责文件下载或分块下载。
3. 下载完成后通过 WS 回 ACK 或状态事件。

因此，长期更合理的组合是：

1. WS 负责主业务总线。
2. HTTP 负责大资源传输。
3. 不再扩展私有裸 TCP 业务协议。

## WS V2 统一消息信封设计

当前 WS 结构只有：

1. `type`
2. `timestamp`
3. `signature`
4. `payload`

这个结构足够支持最小演示，但不足以承载后续完整业务，因为缺少：

1. 唯一消息 ID。
2. 请求与响应关联 ID。
3. 路由信息。
4. 错误结构。
5. 协议版本。
6. 消息交互语义分类。

### 一、推荐字段结构

```ts
export type MessageKind =
  | "COMMAND"
  | "EVENT"
  | "REQUEST"
  | "RESPONSE"
  | "ACK";

export interface MessageRoute {
  role: "teacher" | "student" | "system";
  nodeId: string;
  studentId?: string;
  examId?: string;
  sessionId?: string;
}

export interface MessageError {
  code: string;
  message: string;
  retryable?: boolean;
  details?: unknown;
}

export interface WsEnvelope<T = unknown> {
  version: "v2";
  kind: MessageKind;
  type: string;
  messageId: string;
  correlationId?: string;
  timestamp: number;
  requiresAck?: boolean;
  source: MessageRoute;
  target?: MessageRoute;
  signature?: string;
  error?: MessageError;
  payload: T;
}
```

### 二、字段职责说明

#### 1. `version`

作用：

1. 为协议演进保留版本能力。
2. 避免教师端与学生端版本不一致时互相误解。

建议：固定使用 `"v2"`。

#### 2. `kind`

作用：

1. 标识消息交互语义。
2. 区分命令、事件、请求、响应、确认。

建议取值：

1. `COMMAND`
2. `EVENT`
3. `REQUEST`
4. `RESPONSE`
5. `ACK`

#### 3. `type`

作用：

1. 标识具体业务动作。
2. 对应具体 handler。

建议消息类型：

1. `HEARTBEAT`
2. `EXAM_PAPER_DISTRIBUTE`
3. `EXAM_START`
4. `EXAM_PAUSE`
5. `EXAM_END`
6. `FORCE_SUBMIT`
7. `ANSWER_SYNC`
8. `STATUS_UPDATE`
9. `CHEAT_ALERT`

#### 4. `messageId`

作用：

1. 唯一标识一条消息。
2. 支持去重、日志追踪、重试与防重放。

建议：

1. 使用 UUID v4 或 ULID。
2. 教师端与学生端各自独立生成。

#### 5. `correlationId`

作用：

1. 让 ACK 或 RESPONSE 指向原始请求。
2. 统一请求与回执关联逻辑。

规则：

1. 主动发送消息时可为空。
2. 回 ACK 或 RESPONSE 时，值为原始 `messageId`。

#### 6. `source` 与 `target`

作用：

1. 明确谁发给谁。
2. 提供统一路由上下文。
3. 支持日志、鉴权、审计与业务过滤。

建议包含：

1. `role`
2. `nodeId`
3. `studentId`
4. `examId`
5. `sessionId`

#### 7. `error`

作用：

1. 统一错误表达。
2. 让失败响应不再依赖临时文本拼接。

建议包含：

1. `code`
2. `message`
3. `retryable`
4. `details`

---

## 项目级协议约束建议

### 1. 路由字段不要继续散落在 payload 中

推荐将以下字段尽量上移到 `source` 或 `target`：

1. `studentId`
2. `examId`
3. `sessionId`

这样做的好处：

1. 路由层不必深入 payload 才能决策。
2. 日志与审计字段更统一。
3. handler 的业务载荷更纯粹。

### 2. 所有可确认消息必须支持 ACK

建议对以下动作启用 `requiresAck`：

1. `EXAM_PAPER_DISTRIBUTE`
2. `EXAM_START`
3. `EXAM_PAUSE`
4. `EXAM_END`
5. `FORCE_SUBMIT`

### 3. 所有 ACK 与 RESPONSE 必须带 `correlationId`

这是后续追踪执行结果、失败重试与状态聚合的基础要求。

### 4. `signature` 可先保留，暂不阻塞协议收敛

当前代码中 `signature` 已存在，但暂未形成完整签名校验闭环。

建议节奏：

1. 第一阶段先统一消息模型。
2. 第二阶段再补 HMAC 或签名校验。

不要让签名实现阻塞协议统一本身。

## Rust 框架推荐

### 一、短期最小改动方案

推荐：`tokio-tungstenite`

适用场景：

1. 当前代码基础上平滑演进。
2. 先统一协议，不大改服务骨架。
3. 快速验证控制动作迁移到 WS 的可行性。

优点：

1. 当前项目已在使用。
2. 迁移成本低。
3. 能快速承接统一信封、ACK 与路由升级。

### 二、中期推荐方案

推荐：`axum`

适用场景：

1. 教师端后续需要更完整的网络服务骨架。
2. 需要同时承接 WebSocket 与 HTTP 资源下载。
3. 需要更清晰的状态注入、中间件、日志与鉴权结构。

优点：

1. HTTP 与 WS 可放在同一服务内。
2. 更适合后续增加下载接口、监控接口与管理接口。
3. 与 Tokio 生态结合紧密。
4. 可维护性高于纯手写 socket 服务。

结论：

1. 短期可以继续用 `tokio-tungstenite`。
2. 中期若进入正式网络层重构，优先落到 `axum`。

### 三、不推荐作为主方案的方向

若坚持统一到裸 TCP，则至少需要：

1. `tokio`
2. `tokio-util`
3. `bytes`
4. 长度分帧协议

但该方向不推荐作为本项目主业务总线方案。

## 落地建议

### 第一阶段

目标：先统一协议模型，不立即删除现有 TCP。

建议事项：

1. 扩展现有 `protocol.ts`、教师端 `protocol.rs`、学生端 `protocol.rs`。
2. 引入 `version`、`kind`、`messageId`、`correlationId`、`source`、`target`、`error`。
3. 将现有 `EXAM_START`、`HEARTBEAT`、`ANSWER_SYNC` 先升级到新信封。

### 第二阶段

目标：将控制动作逐步迁移到 WS。

建议优先顺序：

1. `EXAM_START`
2. `EXAM_PAUSE`
3. `EXAM_END`
4. `FORCE_SUBMIT`
5. `EXAM_PAPER_DISTRIBUTE`

### 第三阶段

目标：明确资源传输边界。

建议：

1. 若试卷仍以 JSON 为主，可继续走 WS。
2. 若后续出现大资源包，则引入 HTTP 下载，不再扩展私有 TCP。

---

## 最终结论

本项目当前并非简单意义上的“协议混乱”，而是处于“TCP 控制链路 + WS 实时链路”的过渡阶段。

但从后续扩展、维护成本与业务一致性看，应尽快收敛为：

1. 首次发现或引导使用轻量 bootstrap 机制。
2. 主业务控制与实时消息统一走 WebSocket。
3. 大资源传输单独由 HTTP 承担。

因此，本方案的明确建议是：

1. 不统一到裸 TCP。
2. 统一主业务协议到 WebSocket。
3. 以 WS V2 信封作为后续消息模型标准。
