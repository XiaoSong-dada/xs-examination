# 2026_03_24_前后端规范差异补齐与迁移计划

## 1. 目标

本计划用于承接旧 `.copilot/AGENTS.md` 与当前 `.github` 规范体系之间仍未补齐的有效差异，重点完成四件事：

1. 逐条比对旧 AGENTS 与当前 `.github` 规范，明确哪些条目应迁移、重写或废弃。
2. 为教师端与学生端前端补齐“目录职责 + 注释要求 + 业务代码落点”规范。
3. 为教师端与学生端 Tauri 后端补齐“层次职责 + 注释要求 + 文件落点”规范。
4. 在 `packages/shared-types` 清除决策成立后，重写当前规范中与共享协议落点相关的过时口径。

## 2. 范围

本计划纳入范围：

1. 对比 [.copilot/AGENTS.md](../../.copilot/AGENTS.md) 与 [.github/copilot-instructions.md](../../.github/copilot-instructions.md)、[.github/instructions](../../.github/instructions) 的有效差异。
2. 补齐教师端与学生端前端目录职责说明，包括 `pages`、`hooks`、`services`、`store`、`types`、`utils`、`components`、`layout`、`router`、`styles`。
3. 补齐教师端与学生端后端目录职责说明，包括 `controllers`、`services`、`repos`、`schemas`、`models`、`network`、`state`、`utils`、`layers`、`db`、`config`。
4. 补齐前后端函数注释规范，明确 TypeScript 使用 JSDoc、Rust 使用 Rustdoc，以及注释至少覆盖哪些字段。
5. 补齐“业务代码必须落到对应职责文件夹”的规范表达。
6. 重写 shared-types 相关口径，移除“当前共享协议统一放 `packages/shared-types`”这一失效前提。

本计划不纳入范围：

1. 立即为全仓库现有函数补齐注释。
2. 立即删除 `.copilot/AGENTS.md` 与 `.copilot/project-dependency-map.md`。
3. 立即修改 package.json、删除 `packages/shared-types` 目录或调整代码 import。
4. 立即重写 PRD、TECH_DESIGN、RESEARCH 等项目级设计文档。

## 3. 当前现状

### 3.1 当前 `.github` 规范偏轻，缺少执行层细节

- [.github/copilot-instructions.md](../../.github/copilot-instructions.md) 已覆盖构建命令、架构边界、业务定位顺序。
- [.github/instructions/frontend-boundaries.instructions.md](../../.github/instructions/frontend-boundaries.instructions.md) 与 [.github/instructions/tauri-backend.instructions.md](../../.github/instructions/tauri-backend.instructions.md) 目前只覆盖少量边界规则。
- 当前 `.github` 体系尚未写明“函数注释必须怎么写”“各文件夹分别负责什么”“业务代码修改应放入哪个职责目录”。

### 3.2 旧 AGENTS 仍有一批有效但未迁移的规则

从 [.copilot/AGENTS.md](../../.copilot/AGENTS.md) 可确认，以下主题仍未被当前 `.github` 体系完整承接：

1. 前端文件命名与目录分工表。
2. Rust 目录分层、DTO/schema 放置与工具复用的细粒度说明。
3. TypeScript / React 的 JSDoc 注释规范。
4. Rust 的 Rustdoc 注释规范。
5. Tailwind 与 Ant Design 的协同约定。
6. SeaORM / repo / migration 的职责边界表达。
7. 禁止事项的明确列表。

### 3.3 当前前端目录职责可以基于真实结构落文档

已确认目录事实：

1. 教师端前端位于 [apps/teacher/src](../../apps/teacher/src)，当前包含 `assets`、`hooks`、`layout`、`pages`、`router`、`services`、`settings`、`styles`、`types`、`utils`。
2. 学生端前端位于 [apps/student/src](../../apps/student/src)，当前包含 `assets`、`components`、`hooks`、`layout`、`pages`、`router`、`services`、`store`、`styles`、`types`。
3. [doc/project_dependency_topology.md](../project_dependency_topology.md) 已经给出前端模块组与快速定位口径，可作为目录职责规范的正式链接来源，而不是在多份规范里重复大段复制。

### 3.4 当前后端层次需要按 teacher 与 student 的真实结构分别表达

已确认目录事实：

1. 教师端后端位于 [apps/teacher/src-tauri/src](../../apps/teacher/src-tauri/src)，已形成 `controllers`、`services`、`repos`、`models`、`schemas`、`network`、`state`、`db`、`config`、`utils`、`crypto` 的较完整层次。
2. 学生端后端位于 [apps/student/src-tauri/src](../../apps/student/src-tauri/src)，当前包含 `commands.rs`、`controllers`、`services`、`schemas`、`network`、`db`、`state`、`utils`、`layers`、`config.rs`，其中 `repos/` 目录当前为空。
3. 学生端 `layers/` 当前至少包含 `config/` 与 `persistence/` 两个子层，这意味着后端规范不能简单照搬教师端的三层目录表述。
4. 若后续规范需要使用“实例层”“映射层”“工具类”等术语，必须先与当前目录结构建立明确映射，避免术语和真实目录脱节。

### 3.5 shared-types 清除决策已成立，但当前规范还未撤口径

已确认事实：

1. [apps/student/package.json](../../apps/student/package.json) 与 [apps/teacher/package.json](../../apps/teacher/package.json) 仍声明 `@xs/shared-types`。
2. 在 `apps/**` 范围内检索 `@xs/shared-types`，仅命中上述依赖声明，没有实际源码 import。
3. [.github/copilot-instructions.md](../../.github/copilot-instructions.md)、[.github/instructions/frontend-boundaries.instructions.md](../../.github/instructions/frontend-boundaries.instructions.md)、[.github/instructions/protocol-and-shared-types.instructions.md](../../.github/instructions/protocol-and-shared-types.instructions.md) 仍沿用“共享协议统一放 `packages/shared-types`”的旧口径。

## 4. 新旧规范差异清单

| 技术栈版本锁定 | 仍写 React 18、AntD 5、Router 6 等旧版本 | 全局规范已按当前仓库重写 | 废弃旧版本锁定，不再迁移 |
| 前端命名约定 | 有页面、组件、hook、service、store 命名表 | 仅保留 `use*.ts`、`*Store.ts`、`*Service.ts` 的简要口径 | 迁移并按当前目录重写 |

| 前端注释规范 | 明确要求函数使用 JSDoc | 当前无此要求 | 迁移为前端 instructions 的显式规则 |

| 安全与反作弊条目 | 含未验证的加密、反作弊、版本里程碑条目 | 当前 `.github` 未迁移 | 保持废弃，除非后续代码事实能验证 |

| 禁止事项 | 旧 AGENTS 有红线列表 | 当前 `.github` 仅零散表达 | 迁移与精简，只保留已验证规则 |

## 5. 已确认决策

1. `packages/shared-types` 的“现行共享层”口径需要从 `.github` 规范中撤回，不再继续传播。
2. 前端与后端函数注释规范属于执行层约束，应从旧 AGENTS 迁入新的 `.github/instructions/` 体系。
3. 目录职责规范不应只写“命名格式”，还应明确“新增或改动业务代码必须落到对应职责目录”。
4. 后端规范必须区分教师端与学生端的真实结构，不能直接用一套静态三层表述覆盖两端。
5. 对于旧 AGENTS 中已失真、未落地或无法从当前代码验证的条目，应明确废弃，而不是继续模糊保留。

## 7. 分阶段实施计划

### 阶段一：撤回 shared-types 旧口径

1. 逐项清点当前 `.github` 体系中仍把 `packages/shared-types` 写成现行共享层的文件。
2. 为共享协议与跨端类型补写新的暂行口径，避免在 shared-types 删除后出现规范真空。
3. 决定 [.github/instructions/protocol-and-shared-types.instructions.md](../../.github/instructions/protocol-and-shared-types.instructions.md) 是重写、拆分还是废弃。

### 阶段二：补齐前端约定规范

1. 为教师端与学生端前端新增或更新专门的 instructions，写明各目录职责与业务代码落点。
2. 把旧 AGENTS 中仍有效的命名约定迁入新规范，但统一按当前目录结构重写。
3. 补入 TypeScript / React 的 JSDoc 规范，至少要求写明函数作用、参数、返回值；若存在异常抛出，再补 `@throws`。
4. 明确页面、hook、service、store、type、component、layout、router、style、utils 之间的默认职责边界。

### 阶段三：补齐后端约定规范

1. 为教师端与学生端后端新增或更新专门的 instructions，明确 controllers、services、repos、schemas、models、network、state、db、config、utils、layers 的职责边界。
2. 把 Rustdoc 注释规范迁入后端规范，至少要求写明函数作用、参数与返回值；若返回 `Result`，需写明主要错误来源。
3. 明确“新增业务逻辑优先落到 service，持久化访问落到 repo，DTO 落到 schema，工具复用落到 utils”的默认路径，同时说明学生端当前 `repos/` 为空的例外边界。
4. 为“实例层 / 映射层 / 工具类”建立与实际目录一致的术语映射，避免规范文本与仓库结构冲突。

### 阶段四：迁移收尾与废弃说明

1. 对照差异清单，标记每条旧 AGENTS 条目是“已迁移”“已重写”“已废弃”还是“待确认”。
2. 必要时在 [.copilot/AGENTS.md](../../.copilot/AGENTS.md) 中补充“历史文档”与“已迁移入口”说明，降低误读风险。
3. 更新相关计划文档，确认 shared-types 清理与规范迁移之间的前后顺序已闭环。

## 8. 风险与边界

1. 教师端与学生端后端结构并不完全一致，若强行写成统一三层模板，会误导实际落点。
2. “所有函数都必须补注释”属于高约束规则，若直接追溯全仓库补齐，执行成本会明显放大；迁移时应明确是“新增/修改代码必须遵守”还是“全量补齐”。
3. shared-types 删除后，若没有同步改写协议落点规范，会让后续跨端改动失去统一入口。
4. 旧 AGENTS 中的安全、反作弊、版本里程碑条目包含未验证内容，若不先筛选就迁移，容易把历史设想写成现行规则。

## 9. 最小验收步骤

1. 仓库中存在一份独立的“前后端规范差异补齐与迁移计划”文档。
2. 文档中已逐条列出旧 AGENTS 与当前 `.github` 规范的主要差异及处理决策。
3. 文档中已明确前端注释规范、后端注释规范、前端目录职责、后端目录职责属于本次迁移范围。
4. 文档中已明确 `packages/shared-types` 相关口径需要从当前规范中撤回。
5. 文档中的实施阶段能够直接指导后续更新 `.github` 规范文件，而不是停留在抽象总结。
