# 学生端按题答案同步与教师端监考进度计划（2026-03-23）

## 目标

在当前“连接考生设备 -> 分发试卷 -> 开始考试”闭环基础上，继续打通“学生端按题保存最新答案 -> 教师端接收最新答案 -> 教师端实时监考展示答题进度”的完整链路，形成以下闭环：

1. 教师端点击“开始考试”后，学生端进入正式答题态。
2. 学生端每次作答时，按题保存该题当前最新答案，而不是保存整份答题快照。
3. 学生端答案先写入本地数据库，保证断网、异常退出后仍可恢复。
4. 学生端在连接可用时，通过现有 WebSocket 链路将最新答案同步到教师端。
5. 教师端接收答案后，将最新答案持久化到数据库。
6. 教师端基于答案持久化结果维护每个 student_exam 的答题进度聚合。
7. 教师端实时监考页与报告页不再使用硬编码 0，而是展示真实答题进度。

本次优先目标是完成“最新答案 + 进度聚合 + 前端展示”的可用闭环，不实现历史答题轨迹回放。

---

## 核心决策

### 一、答案存储形式

本次采用“按题存最新答案”的形式，不采用“每次下一题保存整份答题快照”的形式。

原因：

1. 当前需求的核心是断点续答、教师端看到实时进度、教师端保存学生当前答案。
2. 这些目标用“一题一条最新答案记录”的模型最直接，也最符合当前已有表结构。
3. 若每次都保存整份答题快照，会产生重复写入、重复传输和后续查询复杂度。
4. 学生端现有 `local_answers` 表和教师端现有 `answer_sheets` 表都天然更适合“最新答案”模型。

### 二、教师端进度聚合方案

本次采用方案 B：新增 `student_exam_progress` 表，作为教师端监考与报告页的主查询来源。

原因：

1. 监考页是高频轮询场景，若每次都从答案表做聚合，后续扩展时会逐渐增重。
2. `student_exam_progress` 可以把“某个考试记录当前已答题数、总题数、进度百分比”稳定沉淀下来。
3. 该表可以与 `answer_sheets` 分层：前者负责“最新答案事实”，后者负责“监考聚合面”。

### 三、试卷快照边界保持不变

学生端 `exam_snapshots` 继续只保存试卷快照，不承担答题结果存储职责。

原因：

1. `exam_snapshots` 当前职责已经清晰，是试卷与考试元数据快照。
2. 若把答案也放进 `exam_snapshots`，会模糊“题目快照”和“作答状态”的边界。
3. 当前项目已有 `local_answers` 与 `sync_outbox`，应优先复用现有结构。

## 范围

### 本次纳入范围

1. 学生端按题保存最新答案到本地数据库。
2. 学生端复用现有 WebSocket 链路发送 `ANSWER_SYNC` 消息到教师端。
3. 教师端接收 `ANSWER_SYNC` 并持久化最新答案。
4. 教师端新增 `student_exam_progress` 表并维护聚合进度。
5. 教师端实时监考页展示真实答题进度。
6. 教师端报告页展示与监考页一致口径的真实答题进度。
7. 答案同步成功后回写学生端本地同步状态。
8. 为断网补发预留 `sync_outbox` 使用路径。

### 本次不纳入范围

1. 不实现“每次作答历史版本”的完整回放。
2. 不新增“整份答题快照历史表”。
3. 不在本次实现自动交卷、强制交卷、暂停考试等控制动作扩展。
4. 不在本次重构阅卷、判分、成绩汇总整套能力。
5. 不在本次做复杂的多端冲突合并，只保证“最新 revision 不被旧 revision 覆盖”。

## 现状结论

### 一、学生端现状

已具备的基础：

1. 已有考试会话表 `exam_sessions`。
2. 已有试卷快照表 `exam_snapshots`。
3. 已有本地答案表 `local_answers`。
4. 已有同步出站队列表 `sync_outbox`。
5. 已有 `send_answer_sync` 命令与 `ANSWER_SYNC` 协议骨架。

当前缺口：

1. 学生端答题页当前只把答案保存在 React 内存中。
2. `local_answers` 与 `sync_outbox` 虽然已有表结构，但未接入实际答题流程。
3. `send_answer_sync` 当前只负责发送消息，不负责本地落库和发送成功后的状态回写。

### 二、教师端现状

已具备的基础：

1. 已有 `answer_sheets` 表结构与模型定义。
2. 已有 WebSocket 服务，且可识别 `ANSWER_SYNC` 消息。
3. 已有监考页与报告页的“答题进度”展示位。

当前缺口：

1. `ANSWER_SYNC` 当前只打印日志，没有持久化。
2. 教师端还没有 `answer_sheets` 对应的 repo/service/controller 完整链路。
3. 当前没有用于监考的聚合表或聚合查询落点。
4. 监考页与报告页当前仍将 `answerProgress` 硬编码为 0。

## 数据模型方案

### 一、学生端复用现有表

#### 1. local_answers

继续复用现有 `local_answers` 表，作为学生端“按题存最新答案”的主表。

用途：

1. 保存当前考试会话下每道题的最新答案。
2. 保存答案 revision。
3. 保存同步状态与最后同步时间。

当前表结构已满足本次需求，无需新增表。

#### 2. sync_outbox

继续复用现有 `sync_outbox` 表，作为答案同步失败后的待补发队列。

用途：

1. 承接 `ANSWER_SYNC` 待发送事件。
2. 支持断网继续作答后恢复补发。
3. 记录失败次数与重试时间。

当前表结构已满足本次需求，无需新增表。

### 二、教师端修改与新增表

#### 1. answer_sheets

教师端继续使用 `answer_sheets` 保存“每题最新答案”，但建议补齐字段，使其适配当前 `student_exams` 语义。

建议通过新 migration 做如下增强：

1. 新增 `student_exam_id TEXT NOT NULL REFERENCES student_exams(id)`。
2. 新增 `revision INTEGER NOT NULL DEFAULT 1`。
3. 新增 `answer_updated_at INTEGER`，表示学生端该答案最后一次修改时间。
4. 新增 `received_at INTEGER NOT NULL`，表示教师端收到该答案的时间。
5. 将唯一约束调整为 `UNIQUE(student_exam_id, question_id)`。

设计原因：

1. 当前教师端主业务实体已经从旧 `students(exam_id 维度)` 过渡到 `student_exams`。
2. 监考、连接状态、发卷、开考都围绕 `student_exam_id` 运转，答案也应挂在相同维度。
3. `revision` 与时间字段有助于防止旧消息覆盖新答案。

#### 2. student_exam_progress

教师端新增聚合表 `student_exam_progress`。

建议字段如下：

1. `student_exam_id TEXT PRIMARY KEY REFERENCES student_exams(id) ON DELETE CASCADE`
2. `exam_id TEXT NOT NULL REFERENCES exams(id) ON DELETE CASCADE`
3. `student_id TEXT NOT NULL REFERENCES students(id) ON DELETE CASCADE`
4. `answered_count INTEGER NOT NULL DEFAULT 0`
5. `total_questions INTEGER NOT NULL DEFAULT 0`
6. `progress_percent INTEGER NOT NULL DEFAULT 0`
7. `last_question_id TEXT`
8. `last_answer_at INTEGER`
9. `updated_at INTEGER NOT NULL`

建议索引：

1. `idx_student_exam_progress_exam_id` on `(exam_id)`
2. `idx_student_exam_progress_student_id` on `(student_id)`

用途：

1. 作为监考页的主查询来源。
2. 作为报告页的答题进度来源。
3. 每次教师端收到最新答案后立即更新。

## 目标链路

### 一、学生端作答与本地落库链路

1. 学生端收到开考指令后进入正式答题态。
2. 学生在题目上进行选择或输入答案。
3. 前端将答案提交给学生端 Rust 命令层。
4. Rust 服务层将该题答案 upsert 到 `local_answers`。
5. 同时将一条 `ANSWER_SYNC` 事件写入 `sync_outbox`。
6. 如果当前 WebSocket 可用，则立即尝试发送到教师端。
7. 发送成功后更新 `local_answers.sync_status` 与 `last_synced_at`。
8. 若发送失败，则保留 `pending` 或 `failed` 状态，等待后续补发。

### 二、教师端接收与聚合链路

1. 教师端 WebSocket 服务收到学生端 `ANSWER_SYNC`。
2. 解析 payload，识别 `exam_id`、`student_id`、`question_id`、`answer`。
3. 校验该学生是否属于当前考试的 `student_exam`。
4. 将该题最新答案 upsert 到 `answer_sheets`。
5. 根据该 `student_exam_id` 当前已答题数与考试总题数，更新 `student_exam_progress`。
6. 监考页和报告页在轮询时直接读取真实进度。

## 关键风险与处理建议

### 一、答案编码不统一

风险：

1. 学生端页面当前更接近选项下标模型。
2. 后续若多选、判断、填空、主观题进入同一链路，纯下标格式会不稳定。

建议：

1. 单选存选项 key，例如 `A`。
2. 多选存 JSON 数组字符串，例如 `["A","C"]`。
3. 判断、填空、主观题按约定文本格式存储。

### 二、旧消息覆盖新答案

风险：

1. 网络重试或乱序可能导致旧答案覆盖新答案。

建议：

1. 使用 `revision` 或 `answer_updated_at` 做覆盖保护。
2. 仅当新消息版本更高时才覆盖教师端最新答案。

### 三、前端只在“下一题”时保存会丢答案

风险：

1. 如果用户选中答案后还没点下一题就闪退，答案可能丢失。

建议：

1. 实际策略采用“选项变化先写本地，点击下一题再补一次同步尝试”。
