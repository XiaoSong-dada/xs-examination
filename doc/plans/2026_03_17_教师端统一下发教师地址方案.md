# 教师端统一下发教师地址方案（2026-03-17）

## 目标

实现以下控制链路，并明确学生端**不允许自行配置教师端地址**：

1. 学生端启动。
2. 教师端启动。
3. 教师端搜索局域网中已启动学生端程序的设备，并写入教师端设备库。
4. 教师端批量选中设备，统一下发教师端地址列表。
5. 学生端收到下发指令后，先清空本地教师端地址表，再批量写入新地址。
6. 学生端选择 `is_master = 1` 的主地址发起 WebSocket 连接。

---

## 结论

该方案**可行**，但有一个必须先明确的前提：

**教师端下发教师地址，不能依赖现有教师端 WebSocket 通道本身。**

原因是：

1. 学生端在拿到教师地址之前，还不知道应该连接哪个教师端。
2. 因此在“地址下发”这一步，学生端和教师端之间必须先有一条**独立控制通道**。
3. 只有地址成功下发并落库后，学生端才可以再通过现有 WebSocket 骨架连接教师端。

所以整体上需要两条链路：

1. **控制链路**：用于设备发现、配置下发、回执确认。
2. **业务链路**：用于学生端连接教师端 WebSocket、发送心跳和答案同步。

当前仓库已经具备业务链路骨架，因此本次新增重点应放在控制链路。

---

## 现有基础

从当前代码和文档来看，已经有以下基础可以复用：

1. 教师端已有设备发现设计文档，定义了“搜索设备”以及“用发现结果替换设备表”的闭环。
2. 教师端已经有 `devices` 表，用于存储设备列表。
3. 教师端已经有 WebSocket 服务骨架，用于接收学生端心跳和答案同步。
4. 学生端已经新增了 `teacher_endpoints` 表，可作为教师端统一下发后的地址落库位置。

这意味着：

1. “发现设备并入库”这一步有现成设计基础。
2. “地址下发后学生端主动连教师端”这一步有现成 WebSocket 客户端骨架。
3. 你现在缺的不是大方向，而是**地址下发前的控制通道实现**。

## 推荐总体架构

推荐采用以下结构：

### 1. 学生端启动时做三件事

1. 初始化本地数据库。
2. 启动局域网发现响应服务。
3. 启动本地控制服务，等待教师端下发配置。

学生端此时**不需要**提供手工配置教师端地址的页面。

### 2. 教师端启动时做三件事

1. 启动现有 WebSocket 服务。
2. 启动局域网搜索能力。
3. 提供“批量下发教师地址”的管理操作。

### 3. 教师端搜索设备

推荐保留此前文档里的思路：

1. 教师端通过 UDP 广播发送发现请求。
2. 学生端监听固定发现端口，收到请求后返回设备信息。
3. 教师端收集响应结果，展示在设备列表中。
4. 用户确认后，将设备结果写入教师端 `devices` 表。

### 4. 教师端批量下发教师地址

这一步不能走现有 WebSocket，而应走一条独立控制通道。

推荐实现为：

1. 学生端监听一个固定 TCP/HTTP 控制端口。
2. 教师端对选中的学生设备逐台发送“教师地址配置下发命令”。
3. 学生端收到后，在单事务内：
   - 清空 `teacher_endpoints`
   - 批量写入新的地址列表
4. 写入成功后，学生端立即读取 `is_master = 1` 的地址并尝试连接。
5. 学生端将处理结果回执给教师端。

---

## 为什么推荐“独立控制端口”

这是整个方案里最关键的架构决策。

如果没有独立控制端口，就会出现循环依赖：

1. 学生端需要先知道教师端地址，才能连教师端 WebSocket。
2. 但教师端又想通过 WebSocket 把教师端地址发给学生端。
3. 这在启动阶段无法成立。

因此必须在学生端保留一个固定入口，例如：

1. UDP：只负责发现。
2. TCP/HTTP：负责可靠地接收配置下发。

推荐分工如下：

1. **UDP 广播端口**：发现学生设备。
2. **TCP/HTTP 固定控制端口**：接收教师端下发的教师地址配置。
3. **WebSocket 端口**：学生端连接教师端后用于业务通信。

这种拆分的好处是：

1. 各链路职责清晰。
2. 启动顺序明确。
3. 后续做失败重试、回执确认、权限校验时不会和考试业务消息混在一起。

## 数据设计建议

### 一. 学生端 `teacher_endpoints` 表

当前表结构已经可以支持最小可用方案：

```sql
CREATE TABLE teacher_endpoints (
    id          TEXT    PRIMARY KEY,
    endpoint    TEXT    NOT NULL,
    name        TEXT,
    remark      TEXT,
    is_master   INTEGER NOT NULL DEFAULT 0,
    last_seen   INTEGER,
    created_at  INTEGER NOT NULL,
    updated_at  INTEGER NOT NULL
);
```

这个表对于当前目标是够用的。

但如果要让教师端统一下发更稳，建议后续再补两个字段：

1. `config_version INTEGER NOT NULL DEFAULT 1`
2. `applied_at INTEGER`

原因：

1. 便于判断是否接收到了旧配置。
2. 便于教师端查询某台设备是否已经应用了最新配置。

如果当前先走最小闭环，可以先不加。

### 二. 教师端 `devices` 表

现有 `devices` 表只有：

1. `id`
2. `ip`
3. `name`

如果你要把“下发状态”“是否已安装学生端”“最后在线时间”这类信息展示到设备列表中，建议后续扩展：

1. `control_port INTEGER`
2. `app_status TEXT`
3. `last_seen_at INTEGER`
4. `last_push_status TEXT`
5. `last_push_error TEXT`

这不是本次闭环的必须项，但对后续运维和排错非常有价值。

## 协议建议

### 一. 设备发现协议

#### 教师端广播请求

```json
{
  "type": "DISCOVER_STUDENT_DEVICES",
  "timestamp": 1710000000000
}
```

#### 学生端响应

```json
{
  "type": "DISCOVER_STUDENT_DEVICES_ACK",
  "timestamp": 1710000000100,
  "payload": {
    "deviceId": "student-device-uuid",
    "ip": "192.168.1.25",
    "name": "学生机-025",
    "controlPort": 18766,
    "dbReady": true,
    "appVersion": "0.1.0"
  }
}
```

学生端只有在以下条件满足时才返回：

1. 程序已启动。
2. 数据库初始化完成。
3. 控制服务已启动。

这样教师端搜到的设备就是“可接收配置”的设备，而不是单纯能 ping 通的机器。

### 二. 下发教师地址协议

推荐新增一类控制消息：`APPLY_TEACHER_ENDPOINTS`。

#### 教师端下发请求

```json
{
  "type": "APPLY_TEACHER_ENDPOINTS",
  "requestId": "uuid-request-id",
  "timestamp": 1710000000200,
  "payload": {
    "configVersion": 1,
    "endpoints": [
      {
        "id": "endpoint-1",
        "endpoint": "ws://192.168.1.10:18888",
        "name": "主教师端",
        "remark": "主机",
        "isMaster": true
      },
      {
        "id": "endpoint-2",
        "endpoint": "ws://192.168.1.11:18888",
        "name": "备用教师端",
        "remark": "从机",
        "isMaster": false
      }
    ]
  }
}
```

#### 学生端处理规则

1. 校验 payload 至少包含 1 个 endpoint。
2. 校验必须且只能有 1 个 `isMaster = true`。
3. 开启事务。
4. 删除 `teacher_endpoints` 全表。
5. 逐条写入新的 endpoint 记录。
6. 提交事务。
7. 取出主地址，立即发起连接。

#### 学生端回执

```json
{
  "type": "APPLY_TEACHER_ENDPOINTS_ACK",
  "requestId": "uuid-request-id",
  "timestamp": 1710000000300,
  "payload": {
    "success": true,
    "connectedMaster": "ws://192.168.1.10:18888",
    "message": "配置已落库并开始连接主教师端"
  }
}
```

如果失败：

```json
{
  "type": "APPLY_TEACHER_ENDPOINTS_ACK",
  "requestId": "uuid-request-id",
  "timestamp": 1710000000300,
  "payload": {
    "success": false,
    "message": "配置写入失败: 数据库事务异常"
  }
}
```

## 最小实现方案

如果目标是尽快跑通，建议按下面最小版本实施。

### 阶段 1：学生端控制通道骨架

学生端新增：

1. `network/discovery_listener.rs`
   - 监听固定 UDP 端口
   - 收到教师端发现请求后返回本机信息
2. `network/control_server.rs`
   - 监听固定控制端口
   - 接收 `APPLY_TEACHER_ENDPOINTS` 指令
3. `services/teacher_endpoint_service.rs`
   - 事务清空并重建 `teacher_endpoints`
4. `adapters/teacher_endpoints.rs`
   - CRUD 封装

### 阶段 2：教师端搜索与下发

教师端新增：

1. `network/device_discovery.rs`
   - 广播发现
   - 收集 ACK
2. `network/student_control_client.rs`
   - 按设备 IP + 控制端口发送配置下发请求
3. `services/device_push_service.rs`
   - 批量下发并汇总结果
4. 命令层新增
   - 搜索设备
   - 批量下发教师地址

### 阶段 3：联动现有 WebSocket 骨架

学生端在配置落库成功后：

1. 读取 `teacher_endpoints` 中 `is_master = 1` 的记录。
2. 直接调用现有 `connect_teacher_ws` 内部能力发起连接。
3. 后续继续复用现有心跳和答案同步流程。

### 学生端

1. `booting`
2. `db_ready`
3. `control_ready`
4. `discovered`
5. `config_applied`
6. `connecting_teacher`

1. `searching_devices`
2. `devices_discovered`

4. `pushing_teacher_endpoints`
5. `push_success_partial`
6. `push_success_all`

这样后续无论是 UI 展示还是日志定位，都比较清晰。

## 关键风险与建议

### 一. 仅靠 IP 标识设备不够稳定

局域网环境里 IP 可能变化，因此教师端设备库最好不要只依赖 IP。

建议学生端发现响应时返回：

1. `deviceId`
2. `ip`
3. `name`

其中 `deviceId` 应尽量是稳定标识，例如首次启动生成并保存在本地。

### 二. 批量下发需要回执

即便你当前不处理重试，也应该至少有回执，不然教师端无法知道：

1. 哪些机器收到了配置。
2. 哪些机器写库成功。
3. 哪些机器已经开始连接主教师端。

### 三. 地址字符串建议统一用 endpoint，不建议只存纯 IP

虽然业务上你说的是“教师端 IP”，但从实现角度建议统一存 `endpoint`：

1. 当前可以是 `ws://192.168.1.10:18888`
2. 后续也可以支持端口变化
3. 后续也可以支持路径，例如 `/ws`

如果只存 IP，那么端口和协议将来还得拆到别的字段里，扩展性会差。

### 四. 安全性可以后补，但接口要预留

当前阶段可以先不做鉴权，但建议控制消息结构里预留：

1. `requestId`
2. `timestamp`
3. `signature`

这样后续做 HMAC 时不用重新改协议格式。

## 不建议的方案

### 一. 不建议让学生端继续保留手工输入教师地址的设置页

这会和“教师端统一配置”目标冲突。

如果保留，也建议只保留开发调试入口，不出现在正式考试流程中。

### 二. 不建议让教师端直接写学生端数据库文件

这种方案虽然理论上可以实现，但要求教师端拥有远程文件系统权限，风险高，部署复杂，也不符合局域网控制模式。

### 三. 不建议把“地址下发”直接塞进现有考试业务 WebSocket 协议里

因为地址下发发生在 WS 建链之前，放进去会形成循环依赖。

---

## 验收标准

1. 学生端启动后，数据库初始化完成并启动控制服务。
2. 教师端搜索设备时，只能发现已启动并已准备好的学生端。
3. 教师端选择若干设备后，可一键下发教师地址列表。
4. 学生端收到下发请求后，可清空本地 `teacher_endpoints` 并重新写入。
5. 学生端写入完成后，可自动连接 `is_master = 1` 的主教师端。
6. 教师端可看到每台设备的下发结果回执。

---

## 最终建议

如果你的核心目标是：

**学生端不允许自己配置教师端地址，必须由教师端统一下发。**

那么最合适的实现方式是：

1. 保留学生端本地 `teacher_endpoints` 表作为最终落库位置。
2. 教师端负责发现设备。
3. 教师端通过独立控制通道批量下发配置。
4. 学生端收到后事务覆盖本地地址表。
5. 学生端自动连接主教师端。

这个方案和你当前仓库的演进方向是兼容的，且不需要推翻现有 WebSocket 骨架，只是补齐了“建链前控制通道”这一块。
