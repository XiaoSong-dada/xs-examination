# 2026-03-22 学生端断线重连计划

## 目标

本计划聚焦学生端在本地已有考试会话与教师端地址缓存的前提下，实现启动恢复展示与持续重连能力，满足以下目标：

1. 学生端每次启动时，从本地 `exam_sessions` 中读取最新一场考试，并将考试标题、考生信息恢复到头部。
2. 学生端每次启动时，从本地 `teacher_endpoints` 中读取 `is_master=1` 的教师端地址，并将教师端 IP 恢复到头部。
3. 学生端每次启动时，若本地存在主教师端地址且存在最近考试会话对应的 `student_id`，则自动尝试连接该教师端 WebSocket。
4. 当首次连接失败时，学生端会持续重试，直到连接成功。
5. 当连接成功后因教师端异常、网络中断等原因断开时，学生端会再次持续重连，直到恢复连接。

本计划当前只落文档，不进入编码阶段。

## 范围

### 包含范围

1. 学生端启动时恢复最新考试会话。
2. 学生端启动时恢复主教师端地址。
3. 学生端头部展示逻辑调整为“本地缓存展示 + 连接状态独立展示”。
4. 学生端 Rust 后端统一自动连接、失败重试、断线重连。
5. 学生端启动恢复与控制链路之间的连接职责收敛。

### 不包含范围

1. 教师端协议改造。
2. 教师端发卷、开考链路改造。
3. `exam_sessions` 与 `teacher_endpoints` 表结构变更。
4. 学生端答题数据同步逻辑调整。

## 当前现状

基于当前代码，学生端已经具备一部分“启动恢复”能力，但还没有真正完成“自动重连”。

### 已有能力

1. 前端启动后会调用当前考试查询接口，读取本地最新考试会话与试卷快照。
2. 头部初始化时会调用教师运行态查询接口，读取主教师端地址与当前连接状态。
3. 设备 IP 也会在头部初始化阶段同步恢复。
4. 学生端收到教师端 `APPLY_TEACHER_ENDPOINTS` 后，会把教师端地址落库，并立即尝试连接一次主教师端 WebSocket。

### 当前缺口

1. 学生端启动时不会主动根据本地 `teacher_endpoints` 自动连接教师端。
2. 当前控制服务只做一次性连接，不做失败重试。
3. 当前 WebSocket 连接成功后，如果后续断开，也没有统一的自动重连逻辑。
4. 头部当前仍把考试标题与考生信息绑定在“已连接教师端”这一条件上，不符合“启动即展示本地缓存”的目标。
5. 当前 WebSocket 断线状态收口不完整，容易出现状态显示与真实连接状态不一致的问题。

## 已确认的产品决策

本次计划基于以下已确认决策展开：

1. 头部采用“启动即展示本地缓存”的口径，即使当前还未连上教师端，也应展示最近考试与考生信息。
2. 头部中的教师端地址只显示 IP，不显示完整 `ws://host:port`。
3. 自动重连职责放在学生端 Rust 后端，前端只负责展示状态，不在 React 层实现重连循环。
4. 启动自动连接时优先复用本地最近 `exam_sessions` 中的 `student_id`，不新增表结构。

## 实施方案

## 阶段一：固化启动恢复语义

这一阶段的目标是先把“本地缓存恢复”和“头部展示口径”统一下来，为后续自动重连打基础。

### 1. 恢复最新考试会话作为头部展示来源

当前学生端已经能通过当前考试查询接口，从本地 `exam_sessions` 中按最近更新时间取出最新考试会话，并写入前端 `examStore`。

这一能力继续保留，不需要新增数据库表或改动数据来源。

要求明确为：

1. 只要本地存在最近一条 `exam_sessions`，头部就可以展示考试标题、学号、姓名。
2. 这一展示不再以当前是否已连上教师端作为前置条件。

### 2. 恢复主教师端地址作为头部展示来源

当前学生端已经能通过教师运行态查询接口，从本地 `teacher_endpoints` 中取出 `is_master=1` 的记录。

这一能力继续保留，不需要新增表结构。

要求明确为：

1. 头部展示教师端地址时，仅显示主教师端地址中的 IP。
2. 当前无主教师端地址时，显示未配置。

### 3. 调整头部展示门控

当前头部展示逻辑是：

1. 考试标题只有在 `teacherConnectionStatus === connected` 时才展示。
2. 学生信息只有在 `teacherConnectionStatus === connected` 时才展示。

这与本次目标不一致。

需要调整为：

1. 考试标题、学生信息是否展示，只取决于本地 `currentSession` 是否存在。
2. 连接状态图标与文字仍单独反映当前 WebSocket 连接状态。
3. 即使当前正在重连或尚未连上，也保留本地缓存展示。

## 阶段二：建立统一自动重连管理器

这一阶段的目标是把当前分散的一次性连接逻辑，收敛成一个学生端 Rust 后端统一管理的重连入口。

### 1. 统一连接职责入口

当前存在多个可能触发连接的入口：

1. 前端通过命令主动调用连接。
2. 学生端控制服务在收到 `APPLY_TEACHER_ENDPOINTS` 后直接连接。

问题在于：

1. 它们都偏向“一次性 connect”，而不是“确保最终连上”。
2. 后续一旦引入重试，很容易出现多个入口并发发起多条重连循环。

因此需要新增一个统一重连管理器，由它负责：

1. 首次连接。
2. 连接失败后的持续重试。
3. 已连接后断线的持续重连。
4. 切换目标 endpoint 时的连接迁移。

涉及位置：

1. [apps/student/src-tauri/src/lib.rs](apps/student/src-tauri/src/lib.rs)
2. [apps/student/src-tauri/src/network/ws_client.rs](apps/student/src-tauri/src/network/ws_client.rs)
3. [apps/student/src-tauri/src/network/control_server.rs](apps/student/src-tauri/src/network/control_server.rs)
4. [apps/student/src-tauri/src/state.rs](apps/student/src-tauri/src/state.rs)

### 2. 明确重连管理器的输入上下文

启动自动连接至少需要两类信息：

1. 主教师端的 `endpoint`
2. 当前学生端应该带去连接的 `student_id`

本计划建议：

1. `endpoint` 继续来自 `teacher_endpoints.is_master`
2. `student_id` 继续来自最近一条 `exam_sessions`

这样可复用当前数据模型，不增加新表和新字段。

如果出现以下场景：

1. 本地有 `teacher_endpoints`
2. 但没有任何 `exam_sessions`

则该次启动仅恢复头部教师端 IP，不自动发起连接，等待后续教师端重新下发完整上下文。

涉及位置：

1. [apps/student/src-tauri/src/services/exam_runtime_service.rs](apps/student/src-tauri/src/services/exam_runtime_service.rs)
2. [apps/student/src-tauri/src/services/teacher_endpoints_service.rs](apps/student/src-tauri/src/services/teacher_endpoints_service.rs)

### 3. 启动阶段挂接自动连接

在学生端应用启动后，应新增后台任务：

1. 读取本地主教师端地址。
2. 读取最近考试会话中的最小连接上下文。
3. 若两者都存在，则交给统一重连管理器开始连接与持续重试。

这一步的目标是让学生端在冷启动后无需人工干预，就能自动恢复教师端连接。

涉及位置：

1. [apps/student/src-tauri/src/lib.rs](apps/student/src-tauri/src/lib.rs)

### 4. 定义初版重试策略

第一版不建议引入复杂指数退避，优先选择简单稳定的策略。

建议策略：

1. 固定间隔重试，例如每 3 秒或 5 秒尝试一次。
2. 无上限重试，直到连接成功或目标连接上下文被替换。
3. 每次重试前都先检查当前是否已经连上相同 endpoint，避免重复连接。

这样可以先快速建立稳定链路，再决定后续是否细化为退避策略。

## 阶段三：补齐断线后的持续重连

这一阶段的目标是让“首次连上之后又断开”的场景也进入统一自动恢复流程。

### 1. 收口断线判定

当前 WebSocket 客户端存在以下问题：

1. writer loop 退出时会清理连接状态。
2. reader loop 出错时只会退出接收循环，但不会统一清理状态。
3. 心跳发送失败时也只是结束当前循环，没有明确收口到统一断线路径。

这会导致连接状态与真实底层状态可能不一致。

因此需要统一规定：

1. 连接建立失败。
2. reader 出错。
3. writer 出错。
4. 心跳发送失败。

以上任一情况都应进入统一“已断开”处理流程，包括：

1. 清理 `ws_sender`
2. 清理 `ws_endpoint`
3. 更新 `ws_connected=false`
4. 发出断线事件
5. 通知重连管理器继续重试

涉及位置：

1. [apps/student/src-tauri/src/network/ws_client.rs](apps/student/src-tauri/src/network/ws_client.rs)
2. [apps/student/src-tauri/src/state.rs](apps/student/src-tauri/src/state.rs)

### 2. 控制服务切换到统一重连入口

当前控制服务在收到 `APPLY_TEACHER_ENDPOINTS` 成功后，会立刻直接调用一次 `connect`。

这一逻辑需要改为：

1. 地址落库成功。
2. 会话预写入或已有会话确认完成。
3. 将主教师端目标地址与 `student_id` 交给统一重连管理器。
4. 由重连管理器负责保证最终连上，而不是只尝试一次。

这样可避免：

1. 初连失败后直接终止。
2. 启动自连与控制服务下发互相抢占连接入口。

涉及位置：

1. [apps/student/src-tauri/src/network/control_server.rs](apps/student/src-tauri/src/network/control_server.rs)

### 3. 增加并发保护

引入统一重连管理器后，必须避免同时存在多条重连循环。

至少要防止以下场景叠加：

1. 启动时自动连接启动一条重连循环。
2. 控制服务收到新配置又启动一条重连循环。
3. 前端若保留手工连接命令，再次启动第三条重连循环。

因此需要在状态层或重连管理器内部增加最小保护信息，例如：

1. 当前目标 endpoint
2. 当前目标 student_id
3. 当前是否已有重连任务在运行

涉及位置：

1. [apps/student/src-tauri/src/state.rs](apps/student/src-tauri/src/state.rs)
2. [apps/student/src-tauri/src/commands.rs](apps/student/src-tauri/src/commands.rs)

## 阶段四：前端状态收口与展示一致性

这一阶段的目标是让前端在自动重连引入后，仍保持状态表达清晰，不把“缓存恢复”和“在线连接”混为一谈。

### 1. 连接状态口径统一

前端继续复用现有事件：

1. `teacher_endpoint_applied`
2. `ws_connected`
3. `ws_disconnected`

要求明确为：

1. 头部缓存展示与连接状态展示分离。
2. `connecting` 表示已知目标地址且正在尝试连接。
3. `connected` 表示已建立 WebSocket。
4. `disconnected` 表示曾有目标但当前未连接。
5. `unknown` 只用于启动早期尚未完成运行态恢复或确实缺少信息的场景。

涉及位置：

1. [apps/student/src/store/deviceStore.ts](apps/student/src/store/deviceStore.ts)
2. [apps/student/src/services/teacherEndpointService.ts](apps/student/src/services/teacherEndpointService.ts)
3. [apps/student/src/layout/AppHeader.tsx](apps/student/src/layout/AppHeader.tsx)

### 2. 页面语义保持稳定

本计划不改动考试业务语义，但需要明确避免出现因重连逻辑引起的误判。

要求保持：

1. 有 `session` 且无 `snapshot` 时，仍停留在等待页。
2. 有 `snapshot` 但考试未开始时，仍显示等待开考。
3. 只有 `session.status === active` 且存在 `snapshot` 时，才进入答题页。

涉及位置：

1. [apps/student/src/App.tsx](apps/student/src/App.tsx)
2. [apps/student/src/store/examStore.ts](apps/student/src/store/examStore.ts)

## 验收标准

本计划实施完成后，应满足以下验收条件：

1. 本地已有 `exam_sessions` 与 `teacher_endpoints(is_master)` 时，学生端冷启动后头部立即显示最近考试标题、学生信息、教师端 IP。
2. 本地已有主教师端地址与最近考试会话的 `student_id` 时，学生端冷启动后无需人工操作即可自动进入连接流程。
3. 教师端暂不可达时，学生端会持续重试，不会只尝试一次后停止。
4. 教师端恢复后，学生端可自动连上，无需重启学生端。
5. 已连接后若教师端断开，学生端会进入再次重连，直到恢复成功。
6. 教师端再次下发新的主地址时，学生端会切换到新的连接目标。
7. 自动重连引入后，头部展示与页面显示语义不出现倒挂或误判。

## 建议实施顺序

为降低风险，建议按以下顺序编码：

1. 先改头部展示口径，完成“启动即展示本地缓存”。
2. 再新增 Rust 后端统一重连管理器。
3. 再把启动自动连接接到应用启动阶段。
4. 再把控制服务的一次性连接切换到统一重连入口。
5. 最后做状态清理、断线收口与联调验证。

## 验证建议

建议在后续编码完成后，至少执行以下校验：

1. 冷启动恢复验证。
2. 首连失败持续重试验证。
3. 连接后教师端异常断开验证。
4. 主教师端地址切换验证。
5. 学生端等待页与答题页语义回归验证。

建议使用当前学生端常规校验命令：

1. 在 `apps/student/src-tauri` 下执行 `cargo check`
2. 在仓库根目录执行 `pnpm --filter @xs/student build`
