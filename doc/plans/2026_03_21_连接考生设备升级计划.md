# 2026-03-21 连接考生设备升级计划

## 目标

本计划聚焦“连接考生设备”链路的升级，分两阶段推进：

1. 第一阶段：在教师端执行“连接考生设备”时，提前把当前考生信息与当前考试信息传入学生端，并写入学生端本地 `exam_sessions` 表；学生端在 WebSocket 真正连上教师端后，在 Header 中展示学生名称、考试标题等信息。
2. 第二阶段：在后续“分发试卷”时，若学生端已存在相同 `exam_id` 的本地会话，则不更新本地 `exam_sessions` 基础信息，但允许按规则更新试卷快照。这一阶段本次只做规划，不在本次执行范围内实现。

## 当前现状

当前“连接考生设备”链路的真实行为如下：

1. 教师端分配页点击“连接考生设备”。
2. 教师端通过 `connect_student_devices_by_exam_id` 按考试查询已分配设备的考生记录。
3. 教师端向学生端下发 `APPLY_TEACHER_ENDPOINTS` 报文。
4. 该报文当前只包含教师端地址配置和 `student_id`，不包含 `exam_id`、`exam_title`、`student_no`、`student_name` 等考试与考生信息。
5. 学生端收到该报文后，只会把教师端地址写入本地 `teacher_endpoints` 表，并尝试主动连接教师端 WebSocket。
6. 学生端本地 `exam_sessions` 当前仍只在后续“分发试卷”时，由 `DISTRIBUTE_EXAM_PAPER -> upsert_distribution` 链路写入。

因此，当前链路还无法满足以下目标：

1. 连接阶段提前建立学生端本地考试会话。
2. 连接成功后在学生端 Header 中显示学生名称、考试标题。

## 本次确认的关键决策

### 第一阶段成功口径

学生端 Header 显示学生名称、考试标题等信息的最终成功口径，不以单纯收到 `APPLY_TEACHER_ENDPOINTS_ACK` 为准，而以学生端 WebSocket 真正连接教师端成功为准。

也就是说：

1. TCP 控制链路 ACK 成功：代表教师端地址下发成功。
2. 学生端 WebSocket 连上教师端：代表最终可展示“连接成功后的 Header 信息”。

### 第一阶段 Header 数据来源

学生端 Header 中的学生名称、考试标题等信息，优先复用 `examStore` 提供的数据，不继续放在 `deviceStore` 中承载。

职责划分建议如下：

1. `examStore`：负责考试与考生会话数据。
2. `deviceStore`：负责设备 IP、教师端地址、连接状态。

### 第二阶段更新规则

当后续“分发试卷”命中相同 `exam_id` 时：

1. 保留本地已有的 `exam_sessions` 基础信息，不更新考试基础信息。
2. 允许按规则更新试卷快照 `exam_snapshots`。

本规则本次先写入计划，不在本次执行中落地。

## 分阶段实施方案

## 阶段一：连接阶段预写入 exam_sessions，并驱动 Header 展示

### 1. 扩展教师端连接设备下发 payload

需要在教师端“连接考生设备”时，把最小考试会话字段一并下发给学生端。

建议新增或补齐的字段包括：

1. `exam_id`
2. `exam_title`
3. `student_id`
4. `student_no`
5. `student_name`
6. `assigned_ip_addr`
7. 考试时间字段，例如 `start_time`、`end_time`

涉及位置：

1. [apps/teacher/src/hooks/useDeviceAssign.ts](apps/teacher/src/hooks/useDeviceAssign.ts)
2. [apps/teacher/src/services/studentService.ts](apps/teacher/src/services/studentService.ts)
3. [apps/teacher/src-tauri/src/controllers/student_exam_controller.rs](apps/teacher/src-tauri/src/controllers/student_exam_controller.rs)
4. [apps/teacher/src-tauri/src/network/student_control_client.rs](apps/teacher/src-tauri/src/network/student_control_client.rs)

目标是让教师端在按考试读取分配记录后，不只知道“给哪台设备下发哪个教师地址”，还知道“这台设备当前对应哪场考试、哪个学生”。

### 2. 同步扩展学生端控制协议定义

教师端和学生端必须共享同一份 `APPLY_TEACHER_ENDPOINTS` 协议字段定义。

涉及位置：

1. [apps/teacher/src-tauri/src/network/student_control_client.rs](apps/teacher/src-tauri/src/network/student_control_client.rs)
2. [apps/student/src-tauri/src/schemas/control_protocol.rs](apps/student/src-tauri/src/schemas/control_protocol.rs)

这一改动是第一阶段所有后续工作的前置条件。

### 3. 在学生端控制服务中新增“连接阶段会话预写入”逻辑

学生端当前在 `APPLY_TEACHER_ENDPOINTS` 分支中只做地址落库和启动 WebSocket，需要新增“连接阶段 exam session 预写入”逻辑。

建议实现方式：

1. 保持 [apps/student/src-tauri/src/services/teacher_endpoints_service.rs](apps/student/src-tauri/src/services/teacher_endpoints_service.rs) 只负责 `teacher_endpoints`。
2. 在 [apps/student/src-tauri/src/services/exam_runtime_service.rs](apps/student/src-tauri/src/services/exam_runtime_service.rs) 中新增一个专门用于连接阶段的写入方法。
3. 在 [apps/student/src-tauri/src/network/control_server.rs](apps/student/src-tauri/src/network/control_server.rs) 的 `APPLY_TEACHER_ENDPOINTS` 分支中，在地址落库成功后执行该方法。

这一步的目标是：

1. 在未发卷前，学生端本地就已经有一条最小可展示的 `exam_sessions` 数据。
2. 此时仍不写入 `exam_snapshots`，因为试卷尚未下发。

### 4. 定义连接阶段 exam_sessions 的状态语义

连接阶段的 `exam_sessions` 不能与发卷后的 `waiting` 或考试中的 `active` 混淆。

建议明确引入一种能表达“已连接但未发卷”的状态语义，例如：

1. 新状态值
2. 或复用现有字段组合，但前提是不会导致前端误判

涉及位置：

1. [apps/student/src-tauri/migrations/0002_create_exam_sessions.sql](apps/student/src-tauri/migrations/0002_create_exam_sessions.sql)
2. [apps/student/src-tauri/src/services/exam_runtime_service.rs](apps/student/src-tauri/src/services/exam_runtime_service.rs)
3. [apps/student/src/store/examStore.ts](apps/student/src/store/examStore.ts)
4. [apps/student/src/App.tsx](apps/student/src/App.tsx)

这一步非常关键，因为一旦连接阶段写入 `exam_sessions`，前端必须能识别“有 session，但还没收到试卷”的状态。

### 5. 调整学生端 Header 的数据来源与展示时机

学生端 Header 当前左侧考试标题来自 `examStore`，但右侧学生信息并没有真正接通。

第一阶段升级后，建议：

1. 学生名称、考试标题统一来自 `examStore` 的 `currentSession`。
2. 教师端地址与连接状态仍来自 `deviceStore`。
3. Header 的最终展示时机以 `ws_connected` 为准。

也就是说：

1. 即使本地 `exam_sessions` 已写入，但 WebSocket 尚未连接成功，也不应把 Header 当成“连接成功后的最终展示态”。
2. 当 `currentSession` 已有值且 `ws_connected` 为真时，再展示教师端传入的学生名称、考试标题等信息。

涉及位置：

1. [apps/student/src/layout/AppHeader.tsx](apps/student/src/layout/AppHeader.tsx)
2. [apps/student/src/store/examStore.ts](apps/student/src/store/examStore.ts)
3. [apps/student/src/store/deviceStore.ts](apps/student/src/store/deviceStore.ts)
4. [apps/student/src-tauri/src/network/ws_client.rs](apps/student/src-tauri/src/network/ws_client.rs)

### 6. 调整学生端页面判定逻辑，避免误进入答题页

学生端当前页面主要通过 `currentSession` 与 `currentSnapshot` 判断是否显示“等待考试开始”或“进入答题页”。

连接阶段写入 `exam_sessions` 后，必须重新分层页面语义：

1. 有 `session` 且无 `snapshot`：说明已连接或已绑定，但尚未收到试卷。
2. 有 `snapshot`：说明试卷已下发。
3. `session.status === active` 且 `snapshot` 存在：说明可以进入正式答题页。

涉及位置：

1. [apps/student/src/App.tsx](apps/student/src/App.tsx)
2. [apps/student/src/store/examStore.ts](apps/student/src/store/examStore.ts)
3. [apps/student/src/services/examRuntimeService.ts](apps/student/src/services/examRuntimeService.ts)

### 7. 第一阶段验收标准

第一阶段完成后，应满足以下验收条件：

1. 教师端点击“连接考生设备”后，学生端收到的 `APPLY_TEACHER_ENDPOINTS` payload 已包含考试与考生信息。
2. 学生端本地数据库中，同时存在：
   1. `teacher_endpoints` 数据
   2. 连接阶段预写入的 `exam_sessions` 数据
3. 此时 `exam_snapshots` 仍为空。
4. 学生端 WebSocket 真正连上教师端后，Header 可显示：
   1. 学生名称
   2. 考试标题
   3. 教师端地址
   4. 连接状态
5. 学生端在“有 session、无 snapshot”时，仍停留在未发卷/等待状态，不误进入答题页。
6. 第一阶段完成后执行一次正常发卷，不破坏原有分发试卷链路。

## 阶段二：同 exam_id 时保护 exam_sessions 基础信息

### 1. 背景

当前学生端 `upsert_distribution` 的更新逻辑主要按 `session_id` 做覆盖更新。

这意味着：

1. 若后续再次发卷且 `session_id` 变化，但 `exam_id` 相同，可能会出现覆盖不符合预期的问题。
2. 第一阶段完成后，连接阶段已经提前创建了本地 `exam_sessions`，因此第二阶段必须明确“发卷时哪些字段可更新，哪些字段不可更新”。

### 2. 第二阶段目标

第二阶段的目标是：

1. 当学生端本地已存在相同 `exam_id` 的 session 时，不覆盖其基础考试信息。
2. 但允许在发卷时补写或更新 `exam_snapshots`。

建议被保护的基础信息包括：

1. `exam_id`
2. `student_id`
3. `student_no`
4. `student_name`
5. `exam_title`
6. `assigned_ip_addr`
7. 基础状态字段

允许更新的内容包括：

1. `exam_snapshots`
2. 试卷元信息与题目快照
3. 与试卷版本相关的字段

### 3. 第二阶段建议改造点

涉及位置：

1. [apps/teacher/src-tauri/src/services/student_exam_service.rs](apps/teacher/src-tauri/src/services/student_exam_service.rs)
2. [apps/student/src-tauri/src/network/control_server.rs](apps/student/src-tauri/src/network/control_server.rs)
3. [apps/student/src-tauri/src/services/exam_runtime_service.rs](apps/student/src-tauri/src/services/exam_runtime_service.rs)
4. [apps/student/src-tauri/migrations/0002_create_exam_sessions.sql](apps/student/src-tauri/migrations/0002_create_exam_sessions.sql)

第二阶段建议在 `upsert_distribution` 进入原有更新逻辑前，先按 `exam_id` 做本地检查，再决定：

1. 是否跳过 `exam_sessions` 基础字段更新
2. 是否允许更新 `exam_snapshots`

### 4. 第二阶段验收标准

第二阶段完成后，应满足以下条件：

1. 学生端本地已存在相同 `exam_id` 的 `exam_sessions` 时，再次发卷不会覆盖基础考试信息。
2. 试卷快照仍可按规则更新。
3. 前端 `examStore` 读取逻辑不出现异常。
4. Header 展示与答题页入口逻辑保持稳定。

## 风险与注意事项

### 1. 连接阶段写 session 后，前端可能误判为已收到试卷

这是第一阶段最大的风险。

必须通过 `currentSnapshot`、会话状态和 WebSocket 状态共同区分：

1. 已连接未发卷
2. 已发卷未开考
3. 已开考

### 2. 不应把学生名称、考试标题继续塞进 deviceStore

若继续由 `deviceStore` 承载业务会话数据，会让网络连接态和考试会话态继续耦合，后续维护成本会更高。

### 3. 第二阶段必须重新梳理按 session_id 与按 exam_id 的边界

当前分发试卷逻辑按 `session_id` 更新，而第二阶段需要保护 `exam_id` 维度的基础信息。这两个维度必须在设计时统一，否则容易出现：

1. 相同考试多条本地会话
2. 发卷覆盖连接阶段预写入数据

### 4. 可能需要补充数据库约束或索引

如果后续发现学生端本地可能存在同 `exam_id` 的多条记录，需要进一步评估是否补充：

1. `exam_id`
2. 或 `exam_id + student_id`

级别的约束或索引。

## 建议执行顺序

1. 先完成第一阶段协议扩展与学生端连接阶段 session 预写入。
2. 再完成 Header 展示与页面状态分层。
3. 验证第一阶段与现有发卷链兼容。
4. 审核无误后，再开始第二阶段 `exam_id` 保护策略。

## 结论

本次计划的核心不是简单“把更多字段塞给学生端”，而是把“连接成功后的网络状态”和“连接阶段建立的考试会话”拆清楚。

第一阶段完成后，系统应具备如下能力：

1. 教师端连接设备时，学生端提前建立最小考试会话。
2. 学生端在 WebSocket 真正连上教师端后，在 Header 中展示学生名称、考试标题等信息。
3. 后续发卷链路继续负责试卷快照下发与考试进入流程。

第二阶段再在此基础上，增加同 `exam_id` 的基础信息保护策略。
