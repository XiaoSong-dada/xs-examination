# 教师端与学生端连接骨架实施计划（2026-03-16）

## 目标

在不引入复杂业务逻辑的前提下，先搭建可运行的通信骨架，满足以下最小闭环：

1. 教师端可启动 WebSocket 服务并记录学生连接状态。
2. 学生端可发起 WebSocket 连接并周期性发送心跳。
3. 学生端可发送答案同步消息（选择题答案），教师端可接收并打印日志。
4. 前端可通过 Tauri 命令触发连接与查看状态（骨架级别，不做完整页面联调）。

## 技术路线

1. 采用 Rust WebSocket（tokio + tokio-tungstenite）作为实时链路。
2. 消息格式采用 JSON，复用现有 `MessageType`/`WsMessage` 结构。
3. 连接发现（mDNS）后续接入，本次先用固定地址/端口打通基础链路。

## 实施步骤

### 阶段 1：协议与状态骨架

1. 教师端补齐 `network/ws_server.rs` 最小服务：
   - 监听端口
   - 接收连接
   - 读取文本消息
   - 识别 `HEARTBEAT`、`ANSWER_SYNC`
   - 更新内存连接状态
2. 教师端扩展 `state.rs`：
   - 保存在线学生最后心跳时间
   - 暴露查询在线列表的基础方法（供命令层调用）

### 阶段 2：教师端启动与命令

1. 在 `lib.rs` 的 `setup` 中异步启动 WS 服务。
2. 新增命令：查询当前连接状态（用于前端验证骨架是否工作）。

### 阶段 3：学生端客户端骨架

1. 学生端新增 `network/` 模块：
   - `protocol.rs`（消息定义）
   - `ws_client.rs`（连接、心跳、发送答案）
2. 学生端扩展 `state.rs`：
   - 保存客户端句柄/连接状态
3. 学生端新增命令：
   - `connect_teacher_ws(url, student_id)`
   - `send_answer_sync(exam_id, student_id, question_id, answer)`

### 阶段 4：编译验证

1. `apps/teacher/src-tauri`: `cargo check`
2. `apps/student/src-tauri`: `cargo check`
3. 验证无阻塞启动、命令可调用、日志可见。

## 本次不做

1. 不实现完整鉴权签名/HMAC 校验。
2. 不实现断线重连队列与本地离线缓存。
3. 不实现前端页面完整交互，只提供命令骨架。

## 验收标准

1. 教师端启动后可监听 WebSocket 端口。
2. 学生端调用连接命令后，教师端日志可见连接建立。
3. 学生端心跳发送后，教师端在线状态可查询。
4. 学生端发送选择题答案后，教师端可接收到 `ANSWER_SYNC` 消息。
5. 两端 `cargo check` 均通过。
